## Two disconnect-path call sites

Both construct a `CVPipelineResult` **directly** (bypassing `CVPipeline.run()`):

1. **Pre-connection wait** ([VisionRunner.java:135-143](photon-core/src/main/java/org/photonvision/vision/processes/VisionRunner.java#L135-L143)):
   while `!frameSupplier.isConnected()`, every 100 ms:
   ```java
   pipelineResultConsumer.accept(new CVPipelineResult(0l, 0, 0, null, new Frame()));
   ```
2. **Post-connect bad-frame path** ([VisionRunner.java:190-195](photon-core/src/main/java/org/photonvision/vision/processes/VisionRunner.java#L190-L195)):
   when both `processedImage.empty()` and `colorImage.empty()`:
   ```java
   // give up without increasing loop count
   // Still feed with blank frames just dont run any pipelines
   frame.release();
   pipelineResultConsumer.accept(new CVPipelineResult(0l, 0, 0, null, new Frame()));
   ```
   (note the question-mark in the line-189 comment "no point in trying to do anything more?" —
   the original author flagged this as uncertain.)

The 5-arg `CVPipelineResult` ctor
([CVPipelineResult.java:39-46](photon-core/src/main/java/org/photonvision/vision/pipeline/result/CVPipelineResult.java#L39-L46))
**never calls `setImageCaptureTimestampNanos`**, so `imageCaptureTimestampNanos` retains its
default value of `0L`.

## Downstream effect (timestamp)

`NTDataPublisher.accept()`
([NTDataPublisher.java:175-187](photon-core/src/main/java/org/photonvision/common/dataflow/networktables/NTDataPublisher.java#L175-L187)):

```java
var captureMicros = MathUtils.nanosToMicros(result.getImageCaptureTimestampNanos());  // = 0
var offset       = NetworkTablesManager.getInstance().getOffset();                    // server-relative-to-coproc, µs
var simplified   = new PhotonPipelineResult(
                       acceptedResult.sequenceID,        // = 0
                       captureMicros + offset,           // = 0 + offset = offset
                       now + offset,                     //  current wpi::nt::Now in server timebase
                       NetworkTablesManager.getInstance().getTimeSinceLastPong(),
                       TrackedTarget.simpleFromTrackedTargets(acceptedResult.targets),
                       acceptedResult.multiTagResult);
ts.resultPublisher.set(simplified, 1024);
// ...
ts.latencyMillisEntry.set(acceptedResult.getLatencyMillis());  // → MathUtils.nanosToMillis(now - 0)
                                                                //   = coproc uptime in ms (!)
ts.hasTargetEntry.set(acceptedResult.hasTargets());            // false
```

So on every disconnect-path publish:
- **`captureTimestampMicros = offset`** on the wire (server timebase). With a sync'd
  TimeSyncServer, `offset` represents server-time-when-coproc-time-was-0, typically a large
  signed integer (potentially tens of seconds to many minutes, depending on relative boot
  times; could even be negative if the coproc booted before the server).
- **`publishTimestampMicros = now + offset`** is *correct* (current server time).
- **`latencyMillisEntry` ≈ coproc uptime in ms** because
  `getLatencyMillis() = MathUtils.nanosToMillis(wpiNanoTime() - 0)`.
- **`sequenceID = 0`** every cycle (loop count is not incremented on this path —
  [VisionRunner.java:192](photon-core/src/main/java/org/photonvision/vision/processes/VisionRunner.java#L192)).
- **`hasTargets() = false`**, so well-behaved consumers drop the result.

## Consumer-side impact

`PhotonPipelineResult.getTimestampSeconds()`
([PhotonPipelineResult.java:177-179](photon-targeting/src/main/java/org/photonvision/targeting/PhotonPipelineResult.java#L177-L179)):

```java
public double getTimestampSeconds() {
    return metadata.captureTimestampMicros / 1e6;
}
```

For a disconnect frame this returns **`offset / 1e6`** seconds.

- On the **robot** (which runs the TimeSync **server**, so `getOffset() = 0` per
  [TimeSyncManager.java:70](photon-core/src/main/java/org/photonvision/common/dataflow/networktables/TimeSyncManager.java#L70)),
  wpi::nt::Now == server time. So `Timer.getFPGATimestamp() - getTimestampSeconds()` ≈
  `Timer.getFPGATimestamp() - (server_time_when_coproc_was_uptime_0) / 1e6` ≈ **seconds
  since the coproc booted, mapped onto the robot's wall clock** — a value of tens of
  seconds, large enough that wpilib's pose estimator will reject it as stale (default
  history buffer is 1.5 s).
- Visible artifacts:
  - PhotonVision UI's "Latency" widget reads coproc uptime instead of "no data" until
    the camera enumerates. Misleading but not load-bearing.
  - Any robot dashboard plotting `captureTimestampMicros` shows a flat-line at `offset`
    every disconnect → big visual dip.
  - A consumer that ignores `hasTargets()` and feeds every result to
    `addVisionMeasurement` would *attempt* to submit a vision measurement with
    `timestampSeconds = offset/1e6`. wpilib drops it as stale and silently no-ops.

## Severity

**SMELL.** Empty targets list prevents the broken timestamp from corrupting any
well-written consumer. But:

1. The published value looks valid (positive int64, plausibly large for a microseconds
   counter). A new consumer that doesn't know to gate on `hasTargets()` first could
   compute nonsense.
2. The latency entry on NT is wrong (coproc uptime), which is visible UX.
3. The fix is a one-liner — assign `imageCaptureTimestampNanos = MathUtils.wpiNanoTime()`
   before publishing, or use a sentinel like `Long.MIN_VALUE`, or simply don't publish
   on the disconnect path.
4. **Not** pre-found per `memory/audit_prefound.md`.

## Adjacent observation (not catalogued)

`sequenceID = 0` is hardcoded on both disconnect-path constructor calls. A robot-side
consumer that uses `sequenceID` to detect "did I miss a frame?" will see seq=0 forever
during disconnect, then jump to whatever the first real frame's seq is — also smelly,
but harmless until a consumer relies on it.
