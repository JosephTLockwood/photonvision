## Units & sign — declared contract

`PnpResult.java:44-54`:
- `bestReprojErr` Javadoc: "Reprojection error of the best solution, **in pixels**"
- `altReprojErr` Javadoc: "If no alternate solution is found, this is bestReprojErr"
- `messages.yaml:69-72`: `bestReprojErr: float64`, `altReprojErr: float64`. (Wire-protocol confirms the field is `float64`; no scaling on the wire.)

OpenCV's `solvePnPGeneric` reprojection error is **RMS pixel error**, always non-negative.
No sign convention issue. ✓

## How each PnP path populates these fields

| Path                              | bestReprojErr value | altReprojErr value | ambiguity value |
| --------------------------------- | ------------------- | ------------------ | --------------- |
| `solvePNP_SQUARE` (IPPE)          | OpenCV `errors[0]`  | OpenCV `errors[1]` | err[0]/err[1]   |
| `solvePNP_SQPNP` (multi-tag)      | OpenCV `error[0]`   | = bestReprojErr    | 0               |
| `estimateRobotPoseConstrainedSolvepnp` | **0 (default-ctor)** | **0** | **0**           |

The constrained-solvePNP path is the anomaly: it returns
[VisionEstimation.java:287-288](photon-targeting/src/main/java/org/photonvision/estimation/VisionEstimation.java#L287-L288):

```java
var pnpresult = new PnpResult();
pnpresult.best = new Transform3d(new Transform2d(ret[0], ret[1], new Rotation2d(ret[2])));
return Optional.of(pnpresult);
```

…leaving every quality-signal field at its default `0`. Confirmed identical in C++ at
[VisionEstimation.cpp:203-210](photon-targeting/src/main/native/cpp/photon/estimation/VisionEstimation.cpp#L203-L210).

Python does **not** implement `CONSTRAINED_SOLVEPNP` at all (no
`estimateRobotPoseConstrainedSolvepnp` in `photon-lib/py/photonlibpy/estimation/visionEstimation.py`).

## Why this matters

`PoseStrategy.CONSTRAINED_SOLVEPNP` is exposed as a public selectable strategy in Java + C++
([PhotonPoseEstimator.java:107](photon-lib/src/main/java/org/photonvision/PhotonPoseEstimator.java#L107),
[PhotonPoseEstimator.h:50](photon-lib/src/main/native/include/photon/PhotonPoseEstimator.h#L50)).
The PoseStrategy Javadoc at
[PhotonPoseEstimator.java:111-113](photon-lib/src/main/java/org/photonvision/PhotonPoseEstimator.java#L111-L113)
says:

> "Internally, the cost function is a sum-squared of pixel reprojection error + (optionally)
> heading error * heading scale factor."

So the optimizer's final cost has a meaningful residual (it's exactly the quantity a
consumer would want as a quality / stddev signal). The JNI bridge
(`ConstrainedSolvepnpJni.do_optimization`) returns only `[x, y, theta]` — the residual cost
is computed inside the solver and discarded.

## Downstream consequences

1. **Cannot gate vision measurements by reprojection error when using
   CONSTRAINED_SOLVEPNP.** A consumer doing
   `if (estimate.targets[0].getPoseAmbiguity() < threshold) addVisionMeasurement(...)`
   has no meaningful threshold to apply for this strategy — `bestReprojErr = 0` looks
   like a perfect fit regardless of how badly the optimizer converged.

2. **Inconsistent with other strategies' quality-signal semantics.** A robot author who
   has tuned their `addVisionMeasurement` thresholds against `LOWEST_AMBIGUITY` results
   gets very different signals when they switch to `CONSTRAINED_SOLVEPNP`.

3. **Failure mode masking.** If the constrained solver converges to a local minimum far
   from the truth (e.g. bad seed pose + multimodal cost function), the user has no in-band
   way to detect it. `do_optimization` returns null on hard solver failures (per
   [VisionEstimation.java:283-284](photon-targeting/src/main/java/org/photonvision/estimation/VisionEstimation.java#L283-L284))
   but a "solver succeeded with high residual" looks indistinguishable from
   "solver succeeded with low residual".

## Adjacent observation — Python port gap

CONSTRAINED_SOLVEPNP is not implemented in Python. A team using photonlibpy can't
select this strategy at all. Per `audit_prefound.md`, this isn't covered by prior work.
Worth noting but it's a coverage divergence, not a correctness bug; not cataloging
separately.

## Verdict

Filing as **SMELL** — the bug is "a quality signal that's available isn't surfaced".
Real impact bounded by the fact that users who care about quality signal can use
`LOWEST_AMBIGUITY` or `MULTI_TAG_PNP_*` instead. But for anyone who chose
CONSTRAINED_SOLVEPNP because they want gyro-fused robot pose, they lose the ability to
filter by quality.

The fix is mostly mechanical:
- Extend `do_optimization` JNI signature to also return the final residual cost.
- Populate `pnpresult.bestReprojErr` from it.
- Same in C++.

Adding sub-check 2.3a for the Python parity gap.
