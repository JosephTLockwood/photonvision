## The narrow question: unit / encoding match

Field name `publishTimestampMicros` declares **┬Ąs**. Wire encoding from
[NTDataPublisher.java:175,187](photon-core/src/main/java/org/photonvision/common/dataflow/networktables/NTDataPublisher.java#L175-L187):

    var now = NetworkTablesJNI.now();                  // ┬Ąs, wpi::nt::Now
    // ...
    publishTimestampMicros = now + offset;             // ┬Ąs in server timebase

So the encoded value is **┬Ąs**. Serialized as int64 little-endian via the same path as
`captureTimestampMicros` (see check 4.1). **Unit/encoding: clean.** No port divergence.

## What I found while looking ŌĆö a JavaŌåöC++/Python `getTimestampSeconds` divergence

While tracing how `publishTimestampMicros` is consumed downstream, I discovered the three
ports compute `getTimestampSeconds()` **differently**:

**Java** ([PhotonPipelineResult.java:177-179](photon-targeting/src/main/java/org/photonvision/targeting/PhotonPipelineResult.java#L177-L179)):

    public double getTimestampSeconds() {
        return metadata.captureTimestampMicros / 1e6;
    }

**C++** ([PhotonPipelineResult.h:102-104](photon-targeting/src/main/native/include/photon/targeting/PhotonPipelineResult.h#L102-L104)):

    wpi::units::second_t GetTimestamp() const {
        return ntReceiveTimestamp - GetLatency();   // (publish - capture) latency
    }

**Python** ([photonPipelineResult.py:43-53](photon-lib/py/photonlibpy/targeting/photonPipelineResult.py#L43-L53)):

    def getTimestampSeconds(self) -> float:
        latency = (self.metadata.publishTimestampMicros - self.metadata.captureTimestampMicros)
        return (self.ntReceiveTimestampMicros - latency) / 1e6

### Numerical comparison

For a "perfect time-sync" scenario:
- Java returns `T_server_capture`.
- C++/Python return `T_robot_receive - latency_coproc = T_server_capture + network_delay`.

The two answers differ by **network delay between coproc and robot**, typically 1ŌĆō5 ms
over Ethernet, larger over Wi-Fi or under congestion.

For an "imperfect time-sync" scenario (offset drift between coproc and server):
- Java: drifts with the offset error.
- C++/Python: independent of the offset (uses only relative latency).

### Same Javadoc/doxygen, different implementations

Both Java and C++ docs say: "Returns the estimated time the frame was taken, **in the
Time Sync Server's time base (wpi::nt::Now)**." Identical wording. C++ even adds the
hint "This is much more accurate than using GetLatency()" (lines 98-99) ŌĆö which is true
of the **C++ implementation** (it's the receive-minus-latency approach), but the
**Java implementation** doesn't even use that approach.

The Python implementation matches C++ semantically.

**Conclusion: Java is the outlier.** Java's `getTimestampSeconds()` directly returns
`captureTimestampMicros / 1e6`, ignoring `ntReceiveTimestampMicros` entirely. C++ and
Python use a robust-to-time-sync-drift formula.

### Impact

- Stage 6 (this is its territory) ŌĆö flagging now since I already did the analysis.
- A team developing on Java and switching to C++ (or vice versa) gets different timestamps
  by ~network_delay.
- A team running with imperfect time-sync gets meaningfully different latency behavior
  between ports.
- The Java approach is more accurate ONLY when time-sync is perfect AND network delay is
  larger than time-sync error.

## Correction to the 1.1 DIVERGENCE finding

The 1.1 finding claimed: *"Python port has no Javadoc-equivalent."* That was wrong.
Python at
[photonPipelineResult.py:14-15](photon-lib/py/photonlibpy/targeting/photonPipelineResult.py#L14-L15)
has the same misleading comment:

    # Image capture and NT publish timestamp, in microseconds and in the coprocessor
    # timebase. As reported by WPIUtilJNI::now.

So Python *does* repeat the "coprocessor timebase" claim ŌĆö same as Java's getter Javadoc,
inconsistent with the actual server-timebase value on the wire. Will append a correction
to FINDINGS.md.

The Python file also adds its own honest warning at
[photonPipelineResult.py:33-34](photon-lib/py/photonlibpy/targeting/photonPipelineResult.py#L33-L34):

    # Python users beware! We don't currently run a Time Sync Server, so these
    # timestamps are in an arbitrary timebase. This is not true in C++ or Java.

This shifts the implicit contract: Python users are told not to trust the absolute value,
which is consistent with Python's `getTimestampSeconds()` using
`ntReceiveTimestamp - latency` (receive-minus-latency, independent of timebase).
