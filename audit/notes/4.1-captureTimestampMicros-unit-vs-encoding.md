## What gets encoded

Serde packs as `int64`:

- C++: `packet.Pack<int64_t>(value.captureTimestampMicros)` ([Serde.cpp:35](photon-targeting/src/generated/main/native/cpp/photon/serde/PhotonPipelineMetadataSerde.cpp#L35)).
- Python: `ret.encodeLong(value.captureTimestampMicros)` ([Serde.py:52](photon-lib/py/photonlibpy/generated/PhotonPipelineMetadataSerde.py#L52)).

No scaling, no sign manipulation. Raw int64 value → 8 wire bytes.

## Byte order on the wire

- Java `decodeLong`
  ([Packet.java:315-330](photon-targeting/src/main/java/org/photonvision/common/dataflow/structures/Packet.java#L315-L330)):
  Bytes 0..7 are LSB→MSB, shifted `<< 0, 8, 16, 24, 32, 40, 48, 56`. **Little-endian.**
- Python `encodeLong` uses `struct.pack("<q", value)`. The `<` prefix is **little-endian**.
- C++ `Packet::Pack<int64_t>` implementation lives in `photon-targeting` Packet.h — by
  schema-version hash equivalence (md5 `ac0a45f686457856fb30af77699ea356` is identical
  across all three ports), the C++ binary encoding must also be little-endian, otherwise
  the wire format would not be interoperable.

All ports use **little-endian int64**. Round-trip preserves the value.

## Schema-version guard

`MESSAGE_FORMAT = "int64 sequenceID;int64 captureTimestampMicros;int64 publishTimestampMicros;int64 timeSinceLastPong;"`

is hashed to `ac0a45f686457856fb30af77699ea356` and embedded in each port's serde. Photon-serde
includes a schema-version handshake — if a future change renames `captureTimestampMicros` →
`captureTimestampNanos` without updating both sides, the md5 hash mismatches and the
deserializer rejects the packet. **Strong defensive coverage against silent unit changes.**
