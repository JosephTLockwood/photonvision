The JNI maps inputs as **RowMajor** then passes to a function expecting **ColMajor**.
Looks like a layout mismatch, but Eigen handles this correctly: when a `Map<RowMajor>` is
passed to a function taking `Matrix<ColMajor>` by value, Eigen performs an element-by-element
copy that translates indices (not raw bytes). The receiving ColMajor matrix has the same
`M(i, j)` values as the source RowMajor matrix.

## Return value parity

`RobotStateMat = Eigen::Matrix<double, 3, 1>`.

JNI:
- On success: copies `result->data()` (3 doubles: x, y, theta) into a `jdoubleArray`.
- On failure: returns `nullptr`.

Java caller ([VisionEstimation.java:283-288](photon-targeting/src/main/java/org/photonvision/estimation/VisionEstimation.java#L283-L288)):

Same conversion logic on both ports. ✓

## Verdict

**Clean.** The JNI contract is consistent:
- Same parameter order Java ↔ C++.
- Layout mismatch (Java's row-major ↔ C++'s column-major) is handled correctly by Eigen's
  layout-aware copy on parameter-pass-by-value.
- Error handling (null on failure) propagates correctly.

The 2.3 SMELL about CONSTRAINED_SOLVEPNP missing reprojection-error reporting still
applies — the JNI faithfully discards the residual cost because the C++ function doesn't
return it. That's a structural API gap, not a JNI contract issue.

## Adjacent observation (not catalogued)

The Eigen Map<RowMajor> → Matrix<ColMajor> pattern works but is **fragile**:
- The comment at casadi_wrapper.h:45 says "Note that casadi is column major, apparently"
  — implying the original author was uncertain about CasADi's expected layout. CasADi
  itself uses column-major (per its documentation), so the comment is correct, just
  worded tentatively.
- If a future refactor changes Java's `SimpleMatrix` to a column-major variant (or
  changes the JNI map type), the layout-aware copy still keeps the math right. But if
  someone "optimizes" the JNI to skip the copy (e.g., by directly mapping as
  `ColMajor`), the math breaks silently because the raw bytes won't match.
- The mitigation would be to standardize on one layout end-to-end (Java pre-transposes
  to column-major before sending, or JNI maps as ColMajor with proper byte ordering).
