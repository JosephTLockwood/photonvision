## Sign verification

PhotonVision's convention from
[TargetCalculations.java:30-33](photon-core/src/main/java/org/photonvision/vision/target/TargetCalculations.java#L30-L33):

> "Yaw is positive right, and pitch is positive up."

So a tag at image-center-right gives positive `getYaw()`; a tag above optical axis gives
positive `getPitch()`.

In WPILib NWU camera frame: forward = +X, left = +Y, up = +Z.

`Rotation3d(roll, pitch, yaw)` constructor uses extrinsic Euler angles (Tait-Bryan
roll→pitch→yaw):

- Pitch is rotation around +Y (left).
- Yaw is rotation around +Z (up).

A `Translation3d(distance, R)` constructs the vector `R * (distance, 0, 0)`.

For positive PV pitch (target above), the code passes `pitch_arg = -pitchRad < 0`.
Rotation about +Y by negative angle, by right-hand rule, rotates +X toward +Z (upward).
→ Direction vector tilts UPward, matching "tag is above optical axis." ✓

For positive PV yaw (target right), the code passes `yaw_arg = -yawRad < 0`.
Rotation about +Z by negative angle rotates +X toward −Y (right in NWU, since +Y is left).
→ Direction vector tilts toward NWU's right (camera's right), matching "tag is right of
optical axis." ✓

## Hand-derivation (synthetic case)

Target straight ahead of camera, 2m away, 0° yaw, 0° pitch:
- Rotation3d(0, 0, 0) → identity.
- Translation3d(2.0, identity) = (2, 0, 0) in NWU camera frame.
- Direction is +X = forward. ✓

Target straight ahead, 30° to the right (PV: yaw = +30°):
- Rotation3d(0, 0, -30°·π/180) = R_z(-π/6).
- R_z(-π/6) * (2, 0, 0) = (2·cos(−π/6), 2·sin(−π/6), 0) = (√3, -1, 0).
- Direction in NWU camera frame: forward + slight right. ✓

Target straight ahead, 30° up (PV: pitch = +30°):
- Rotation3d(0, -30°·π/180, 0) = R_y(-π/6).
- R_y(-π/6) * (2, 0, 0) = (2·cos(−π/6), 0, -2·sin(−π/6)·(−1)) = (√3, 0, +1).
  (R_y matrix is [cos, 0, sin; 0, 1, 0; -sin, 0, cos]; for θ=−π/6: [√3/2, 0, -1/2; 0, 1, 0; 1/2, 0, √3/2]. Applied to (2, 0, 0) gives (√3, 0, 1). +Z is up.)
- Direction in NWU camera frame: forward + slight up. ✓

Signs are correct.

## Frame conversion

`camToTagTranslation.rotateBy(robotToCamera.getRotation())` — WPILib's `Translation3d.rotateBy(R)`
performs the active rotation `R * v`. Since `robotToCamera.getRotation()` represents the
camera's orientation expressed in the robot's frame (i.e., the rotation matrix whose
columns are the camera's basis vectors in robot coords), applying it to a camera-frame
vector gives the same vector in robot coords. **Correct frame conversion.** ✓

Then `.rotateBy(headingSample)` (a Rotation2d) takes the robot-frame planar vector and
rotates it by the robot's heading to express it in the field frame. ✓

## Cross-port comparison

All three ports compute the same formula:
- Java uses `Math.toRadians(deg)`.
- C++ uses `wpi::units::degree_t(deg)` and lets WPI's units-library auto-convert to radians.
- Python uses `wpimath.units.degreesToRadians(deg)`.

Math is byte-for-byte equivalent. **No port divergence on the trig math itself.**

## Found while looking: Python `EstimatedRobotPose` lacks the `strategy` field

This is **not** a sign-convention issue but is a real cross-port API divergence in the
same code area.

[photonPoseEstimator.py:189-191](photon-lib/py/photonlibpy/photonPoseEstimator.py#L189-L191)
and similar call sites at lines 213, 257:

    return EstimatedRobotPose(
        Pose3d(robotPose), result.getTimestampSeconds(), result.getTargets()
    )

Three arguments. The Python `EstimatedRobotPose` dataclass at
[estimatedRobotPose.py:25-36](photon-lib/py/photonlibpy/estimatedRobotPose.py#L25-L36)
has only three fields: `estimatedPose`, `timestampSeconds`, `targetsUsed`. There's no
`strategy` field.

Compare Java
([PhotonPoseEstimator.java:332](photon-lib/src/main/java/org/photonvision/PhotonPoseEstimator.java#L332)):
`new EstimatedRobotPose(..., PoseStrategy.PNP_DISTANCE_TRIG_SOLVE)` — 4 args.

C++
([PhotonPoseEstimator.cpp:397-399](photon-lib/src/main/native/cpp/photon/PhotonPoseEstimator.cpp#L397-L399)):
`EstimatedRobotPose{..., PNP_DISTANCE_TRIG_SOLVE}` — same.

Python users can't inspect `estimate.strategy` to find out which strategy produced a
pose. If they use a top-level `update()` dispatcher (per the docs comment at
[photonPoseEstimator.py:44-45](photon-lib/py/photonlibpy/photonPoseEstimator.py#L44-L45)
"using the strategy set below"), they have no way to confirm which strategy fired.

Severity: **DIVERGENCE** (medium-low). Filing this. Impact is "Python users miss
introspection feature" — not a correctness bug per se, but a deliberate API mismatch
across ports.
