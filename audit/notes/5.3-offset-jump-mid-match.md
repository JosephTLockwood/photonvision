## Re-fetch model

Each call to `NTDataPublisher.accept()` reads a fresh `offset`:

    var now = NetworkTablesJNI.now();
    var captureMicros = MathUtils.nanosToMicros(result.getImageCaptureTimestampNanos());
    var offset = NetworkTablesManager.getInstance().getOffset();   // ← fresh read per frame
    ...
    captureTimestampMicros = captureMicros + offset;
    publishTimestampMicros = now + offset;

No smoothing, hysteresis, or rate-limit at the PhotonVision layer. The published
timestamp moves in lock-step with whatever the underlying `TimeSyncClient.getOffset()`
returns at that instant.

## Failure modes (and where they're covered)

### (a) Initial sync / NT reconnect — offset jumps from 0 → real value

Covered by the 5.1 SMELL: during the ~1–2 s window after startup / NT reconnect,
`getOffset()` returns 0 because the JNI client hasn't received a pong yet. After the
first pong, offset jumps by the full coproc-vs-server boot delta (potentially many
seconds). The "publish-side timestamps abruptly become correct" transition is the same
issue from the consumer's POV.

### (b) Periodic small adjustments — offset drifts by ~µs to ms

WPI's TimeSyncClient applies its own filtering (the JNI is external code, not auditable
here). Between consecutive frames at 60 fps (~16.7 ms apart), the offset typically moves
sub-millisecond per update. The effect on consecutive `captureTimestampMicros` values is
sub-millisecond, well below the 60 fps frame period. **No catalog-worthy issue.**

### (c) Monotonicity edge case — non-monotonic `captureTimestampMicros`

The contract on `captureTimestampMicros` does **not** require monotonicity (only
`sequenceID` is documented as "monotonically increasing" at
[PhotonPipelineMetadata.java:30-31](photon-targeting/src/main/java/org/photonvision/targeting/PhotonPipelineMetadata.java#L30-L31)).
A large negative offset adjustment between consecutive frames CAN cause the published
captureTimestampMicros to decrease:

    Frame N:  capture_N = 1000.0 ms (local); offset_N = +100 ms → publish 1100.0 ms.
    Frame N+1: capture_{N+1} = 1016.7 ms;    offset_{N+1} = -100 ms → publish 916.7 ms.

A robot-side consumer reading these two timestamps sees frame N+1 appear to have
happened *before* frame N. WPILib's `TimeInterpolatableBuffer` can tolerate
out-of-order inserts up to a point, but the pose estimator's idiomatic
`addVisionMeasurement(estimate, timestamp)` doesn't take well to time-going-backward —
it would interpolate against the wrong odometry sample, producing a transient
pose-estimate glitch.

In practice this only matters when offset changes by more than the inter-frame interval.
That requires either initial sync (covered by 5.1) or a major time-sync re-convergence
(also covered by 5.1, since `reportNtConnected` recreates the client).

### (d) Cross-port amplification with the 5.2 BLOCKER

In C++ specifically, `GetTimestamp() = value.time − 2·latency` (5.2 finding). The
`value.time` is NT4's send-time on the wire (which already includes the offset
correction). If offset jumps by Δ between frames N and N+1:

- Frame N: published `captureTimestampMicros = capture + offset_N`. NT4 timestamps the
  send by Δ less than it would have post-jump.
- Frame N+1: published with `+ offset_{N+1} = offset_N + Δ`. NT4 send-time accordingly
  shifted.

C++ then computes `GetTimestamp = value.time − 2·latency`. The Δ shift propagates
through both terms but `value.time` shifts by Δ and `latency` is unchanged. So
`GetTimestamp` shifts by Δ too. Same magnitude as Java/Python — but layered on top of
the systematic −latency offset (5.2). So C++ users observe **the systematic offset PLUS
the offset-jump transient** during NT reconnect.

## Verdict

No new finding for 5.3. The behaviors are already documented by:
- **5.1 SMELL** — transient offset==0 window during startup / NT reconnect.
- **5.2 BLOCKER** — systematic latency double-subtraction in C++.

5.3's analysis adds the **non-monotonicity observation** (large offset adjustments can
cause captureTimestampMicros to go backward), but this is the same root cause as 5.1
(the offset is fetched fresh per-frame with no smoothing or hysteresis).

## Adjacent observation (not catalogued)

WPILib's `TimeSyncClient.getOffset()` JNI is external code. PhotonVision's design
inherits whatever filtering / smoothing properties that JNI provides. If the JNI's
internal filter is replaced or tuned, PhotonVision's behavior changes without source
changes here. This is a coupling-to-upstream concern, not a PhotonVision bug. NIT.
