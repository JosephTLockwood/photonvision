## Conversion matrices — all three ports identical

Both directions are stored as 3×3 rotation matrices:

    NWU_TO_EDN = [  0  -1   0
                    0   0  -1
                    1   0   0 ]

    EDN_TO_NWU = [  0   0   1
                   -1   0   0
                    0  -1   0 ]

Byte-identical across Java
([OpenCVHelp.java:61-62](photon-targeting/src/main/java/org/photonvision/estimation/OpenCVHelp.java#L61-L62)),
C++ ([OpenCVHelp.h:39-42](photon-targeting/src/main/native/include/photon/estimation/OpenCVHelp.h#L39-L42)),
and Python ([openCVHelp.py:12-13](photon-lib/py/photonlibpy/estimation/openCVHelp.py#L12-L13)).

### Sanity-check the matrices

`NWU_TO_EDN @ v_NWU = v_EDN` should hold. Columns of `NWU_TO_EDN` are the NWU basis
vectors expressed in EDN coordinates:

- Col 0 (NWU +X = North) → EDN frame: North is EDN's +Z. → `(0, 0, 1)^T`. ✓
- Col 1 (NWU +Y = West) → EDN frame: West = −East = EDN's −X. → `(-1, 0, 0)^T`. ✓
- Col 2 (NWU +Z = Up)   → EDN frame: Up = −Down = EDN's −Y. → `(0, -1, 0)^T`. ✓

`EDN_TO_NWU` is the transpose / inverse of the above. ✓

Both are proper rotation matrices: det = +1, orthogonal. **Handedness is preserved**
(both NWU and EDN are right-handed). No reflection.

## Rotation conjugation helpers

Java `rotationEDNtoNWU`
([OpenCVHelp.java:270-272](photon-targeting/src/main/java/org/photonvision/estimation/OpenCVHelp.java#L270-L272)):

    return EDN_TO_NWU.inverse().rotateBy(rot.rotateBy(EDN_TO_NWU));

C++ `RotationEDNToNWU`
([OpenCVHelp.h:187-190](photon-targeting/src/main/native/include/photon/estimation/OpenCVHelp.h#L187-L190)):

    return EDN_TO_NWU.Inverse().RotateBy(rot.RotateBy(EDN_TO_NWU));

Python `rotationEDNToNWU`
([openCVHelp.py:144-150](photon-lib/py/photonlibpy/estimation/openCVHelp.py#L144-L150)):

    return -EDN_TO_NWU + (rot + EDN_TO_NWU)

WPILib's `Rotation3d.rotateBy(other)` computes `other.q * self.q` (quaternion multiplication,
extrinsic). So in all three ports:

    rotationEDNtoNWU(R_EDN) = EDN_TO_NWU * R_EDN * EDN_TO_NWU^{-1}

This is the **standard similarity transformation** for re-expressing a rotation under a
basis change `v_NWU = EDN_TO_NWU @ v_EDN`. Mathematically correct. ✓

## Translation conversion

Each port: `trl.rotateBy(EDN_TO_NWU)` (or `RotateBy`). A translation is a vector, so
re-expressing it under a basis change is just multiplying by the change-of-basis matrix.
Direct, no conjugation needed. ✓

## End-to-end SQPnP path (multi-tag)

VisionEstimation.estimateCamPosePNP multi-tag branch
([VisionEstimation.java:141-154](photon-targeting/src/main/java/org/photonvision/estimation/VisionEstimation.java#L141-L154)):

1. `objectTrls` built from `tagModel.getFieldVertices(tag.pose)` — in **NWU field frame**.
2. Passed to `solvePNP_SQPNP`.
3. `solvePNP_SQPNP` internally calls `TranslationToTVec` which applies
   `TranslationNWUtoEDN` per-vector before handing to OpenCV (EDN).
4. OpenCV solves in EDN, returns rvec/tvec in EDN.
5. `TVecToTranslation` / `RVecToRotation` apply `*EDNtoNWU` to convert back to NWU.

Round-trip: **NWU input → EDN solver → NWU output.** Bidirectional conversion is
balanced and identical across ports. ✓

Same pattern in C++
([OpenCVHelp.h:278-...](photon-targeting/src/main/native/include/photon/estimation/OpenCVHelp.h#L278))
and Python (corresponding `solvePNP_SQPNP` function in `openCVHelp.py`).

## Parallel implementation in MathUtils — a SMELL

`MathUtils.convertOpenCVtoPhotonTransform`
([MathUtils.java:178-181](photon-core/src/main/java/org/photonvision/common/util/math/MathUtils.java#L178-L181))
uses **WPILib's** `CoordinateSystem.convert(transform, EDN(), NWU())` rather than PV's
own `EDN_TO_NWU` matrix.

These two paths are exercised side-by-side:
- `OpenCVHelp.solvePNP_*` (PV matrix path).
- `AprilTagPipeline.java:226-234` / `ArucoPipeline.java:215-217` / `SolvePNPPipe.java:101`
  (WPILib-library path).

The test `testOpenCVToPhoton`
([CoordinateConversionTest.java:67-79](photon-core/src/test/java/org/photonvision/common/util/CoordinateConversionTest.java#L67-L79))
asserts the WPILib path produces `Translation3d(3, -1, -2)` for input `(1, 2, 3)` — which
matches PV's matrix output `EDN_TO_NWU @ (1, 2, 3)^T = (3, -1, -2)`. So numerically they
agree today. But maintaining two parallel implementations is a maintenance smell: if
WPILib silently changes its `CoordinateSystem` semantics in a future release, only one
code path adjusts and the two paths diverge.

This is a NIT, not catalog-worthy without observable divergence. Listed here as an
adjacent observation only.
