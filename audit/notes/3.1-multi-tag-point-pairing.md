Both `knownTags` and `corners` are populated **inside the same `ifPresent` lambda**, so
tags whose pose is missing from the layout are dropped from BOTH lists in lockstep.
After the loop:
- `knownTags[k]` = the k-th tag whose pose is in the layout.
- `corners[4k .. 4k+3]` = the same tag's 4 detected corners.
- `objectTrls[4k .. 4k+3]` = the same tag's 4 field vertices (in `TargetModel` vertex order).

So tag-level grouping is preserved. The question reduces to: within a single tag's 4
slots, does `objectTrls[4k+i]` correspond to the same physical corner as `points[4k+i]`?

## Within-tag corner order

### Object vertices — `TargetModel(width, height)` ctor ([TargetModel.java:64-73](photon-targeting/src/main/java/org/photonvision/estimation/TargetModel.java#L64-L73))

    new Translation3d(0, -W/2, -H/2),   // vertices[0]
    new Translation3d(0,  W/2, -H/2),   // vertices[1]
    new Translation3d(0,  W/2,  H/2),   // vertices[2]
    new Translation3d(0, -W/2,  H/2));  // vertices[3]
    // comment line 67: "this order is relevant for AprilTag compatibility"

In tag-local NWU (X=normal, Y=tag's-own-left, Z=tag's-own-up), mapping each to image
position when the tag faces the camera:

- Tag-local +Y is "tag's left", which appears on the **camera's right** (mirror); thus
  `+Y → image +x`.
- Tag-local +Z is "tag's up", which appears at the **camera's top**; thus `+Z → image -y`.

Mapping:
- vertices[0] `(Y=-W/2, Z=-H/2)` → image `(-x, +y)` = **bottom-left (BL)**
- vertices[1] `(Y=+W/2, Z=-H/2)` → image `(+x, +y)` = **bottom-right (BR)**
- vertices[2] `(Y=+W/2, Z=+H/2)` → image `(+x, -y)` = **top-right (TR)**
- vertices[3] `(Y=-W/2, Z=+H/2)` → image `(-x, -y)` = **top-left (TL)**

Object order = **[BL, BR, TR, TL]** (CCW from BL).

## Verdict

Index pairing is preserved end-to-end:
- Object 3D points and image 2D points enter `solvePNP_SQPNP` in matching order.
- `solvePNP_SQPNP` itself

## Adjacent concerns (not cataloging)

- **No assertion that each tag yields exactly 4 corners.** The defensive check is
  `corners.size() % 4 != 0`
  ([VisionEstimation.java:115](photon-targeting/src/main/java/org/photonvision/estimation/VisionEstimation.java#L115)),
  which catches gross misalignment but not the case "tag A: 4 corners, tag B: 0 corners,
  tag C: 4 corners" (total 8, % 4 = 0, passes — but `objectTrls` has 12 entries,
  `points` has 8 → misalignment). In production this is unreachable because both
  AprilTag and ArUco TrackedTarget constructors always set exactly 4 corners; only
  hand-constructed PhotonTrackedTargets (test code) could trigger it. **NIT, not
  cataloging.**

- **The "tag-local +Y maps to image right" mapping** above only holds when the tag is
  upright facing the camera. For arbitrary tag orientation, the per-corner mapping is
  determined by the actual 3D→2D projection geometry — the **physical identity** of
  each corner is what matters, not its image position. The comment in `TargetModel.java:67`
  ("this order is relevant for AprilTag compatibility") encodes that the model's
  vertex[i] is the same physical corner of the tag as the detector's corner[i].
