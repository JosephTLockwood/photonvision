## End-to-end trace (Java V4L2 USB camera path)

Suppose a V4L2 buffer is dequeued with `v4l2_buffer.timestamp = 12_345_678_000 µs`
(wpi::Now monotonic axis, microseconds since HAL/PV init).

1. **Kernel V4L2 driver** (external): writes `v4l2_buffer.timestamp` when the buffer transitions
   QUEUED → DONE. Default anchor is `V4L2_BUF_FLAG_TSTAMP_SRC_EOF` (end-of-frame readout).
   Drivers can opt in to `V4L2_BUF_FLAG_TSTAMP_SRC_SOE` (start-of-exposure) but most UVC cameras
   do not. **No exposure-duration / readout-time correction is applied anywhere in
   PhotonVision.** This is exactly what PR #2489 + driver #33 (mid-exposure anchor) address.
2. **cscore `UsbCameraSourceImpl`** (external): rebases the V4L2 buffer timestamp onto
   `wpi::Now()` (CLOCK_MONOTONIC since HAL init, µs). Stores in `cs::Frame::time`.
3. **PhotonVision USBFrameProvider.getInputMat()** —
   [USBFrameProvider.java:75](photon-core/src/main/java/org/photonvision/vision/frame/provider/USBFrameProvider.java#L75)
   (`m_blockForFrames` path) or
   [USBFrameProvider.java:100](photon-core/src/main/java/org/photonvision/vision/frame/provider/USBFrameProvider.java#L100)
   (`grabRawSinkFrameTimeoutLastTime` path). Both call cscore's grabFrame variant, which
   returns the cs::Frame::time in **µs**, then multiply by 1000 → **ns**. Wraps as
   `CapturedFrame.captureTimestamp = 12_345_678_000_000 ns` (wpi::Now timebase).
   - Inline comment claims: "This is from wpi::nt::Now, or WPIUtilJNI.now()". Implicit
     assumption that these two clocks are identical — they are by default (NT defaults its
     `nt::Now()` to `wpi::Now()` on the client side), but this is a load-bearing assumption
     for NT4 time-sync. Flagged as adjacent concern (Stage 5).
4. **CpuImageProcessor.get()** —
   [CpuImageProcessor.java:88-96](photon-core/src/main/java/org/photonvision/vision/frame/provider/CpuImageProcessor.java#L88-L96).
   Passes `input.captureTimestamp` through to `new Frame(seq, ..., 12_345_678_000_000 ns, ...)`.
   `Frame.timestampNanos` carries the value verbatim.
5. **CVPipeline.run()** —
   [CVPipeline.java:82](photon-core/src/main/java/org/photonvision/vision/pipeline/CVPipeline.java#L82).
   `result.setImageCaptureTimestampNanos(frame.timestampNanos)`. CVPipelineResult carries
   `imageCaptureTimestampNanos = 12_345_678_000_000 ns` (wpi::Now timebase).
6. **NTDataPublisher.accept()** —
   [NTDataPublisher.java:175-190](photon-core/src/main/java/org/photonvision/common/dataflow/networktables/NTDataPublisher.java#L175-L190).
   - `now = NetworkTablesJNI.now()` — current wpi::nt::Now (µs).
   - `captureMicros = MathUtils.nanosToMicros(result.getImageCaptureTimestampNanos())` →
     `12_345_678_000 µs` (wpi::Now base).
   - `offset = NetworkTablesManager.getOffset()` — Time-Sync-Server-vs-local offset.
   - Builds `PhotonPipelineResult(seq, captureMicros + offset, now + offset, timeSinceLastPong,
     simpleTargets, multiTagResult)`.
7. **PhotonPipelineResult ctor** —
   [PhotonPipelineResult.java:67](photon-targeting/src/main/java/org/photonvision/targeting/PhotonPipelineResult.java#L67).
   Builds `PhotonPipelineMetadata` with `captureTimestampMicros = captureMicros + offset`
   (Time-Sync-Server timebase, µs).
8. **Serde** —
   [PhotonPipelineMetadataSerde.cpp:43](photon-targeting/src/generated/main/native/cpp/photon/serde/PhotonPipelineMetadataSerde.cpp#L43),
   [PhotonPipelineMetadataSerde.java:80](photon-targeting/src/generated/main/java/org/photonvision/struct/PhotonPipelineMetadataSerde.java#L80),

## Anchoring summary

Concrete value `captureTimestampMicros` on the wire represents:
**`wpi::Now()`-rebased V4L2 buffer timestamp + server-offset**, where the V4L2 buffer
timestamp is **end-of-frame-readout** (default) or start-of-exposure (only if driver opts in
to `V4L2_BUF_FLAG_TSTAMP_SRC_SOE`). For the libcamera path, the corresponding anchor is
whatever libcamera reports as `SensorTimestamp` (end of exposure on most CSI sensors;
**not** start, not mid).

**Mid-exposure correction is not applied.** This matches `memory/audit_prefound.md` —
addressed by in-flight PR #2489 (PV) + driver #33. Not re-cataloging as a finding.

## Adjacent concerns surfaced

These are not cataloged here as findings; they're added to PROGRESS.md as sub-checks for
deeper future iterations:

- **A1 — Doc divergence on `getCaptureTimestampMicros()` getter Javadoc** vs the field
  comment. The field comment says "wpi::nt::Now on the time sync server" but the getter
  Javadoc claims "the coprocessor's time base". The code at NTDataPublisher.java:186 adds
  the server offset, so the published value is server-timebase, not coproc-timebase. The
  getter Javadoc is wrong. → Catalog as DIVERGENCE in FINDINGS.md.
- **A2 — VisionRunner publishes `new Frame()` (default timestampNanos=wpiNanoTime() at
  construct time but never bound to any captured image) on disconnect/spurious paths** at
  [VisionRunner.java:137](photon-core/src/main/java/org/photonvision/vision/processes/VisionRunner.java#L137)
  and
  [VisionRunner.java:195](photon-core/src/main/java/org/photonvision/vision/processes/VisionRunner.java#L195).
  Since `CVPipelineResult` is built bypassing `CVPipeline.run()`, no
  `setImageCaptureTimestampNanos` call fires; `imageCaptureTimestampNanos` defaults to 0,
  so NT publishes `captureTimestampMicros = 0 + offset`. A robot reading this gets
  what looks like a frame captured at "the time-sync-server epoch" (effectively far in the
  past) — `getTimestampSeconds()` returns a number derived from `offset`, not zero.
  Targets list is null/empty so a well-written consumer drops it, but it's a smelly
  published value. Add as new sub-check 1.3 in PROGRESS.md.
- **A3 — wpi::Now() vs wpi::nt::Now() identity assumption** in
  USBFrameProvider's comment. Folded into existing Stage 5 sub-checks.
- **A4 — LibcameraGpu race**: `MathUtils.wpiNanoTime() - (libcameraNow - libcameraCapture)`
  is computed across two non-atomic clock samplings at
  [LibcameraGpuFrameProvider.java:86-100](photon-core/src/main/java/org/photonvision/vision/frame/provider/LibcameraGpuFrameProvider.java#L86-L100).
  Small positive bias possible. NIT — not actionable without runtime data. Note only.
