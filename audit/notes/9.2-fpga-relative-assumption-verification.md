# Check 9.2 — Verify example projects treat `getTimestampSeconds()` as FPGA-relative

- Iteration: 2026-05-16T18:55-04:00
- Audit root commit: `0e834f0851584237b0dd0913381b52fe6110dd08`
- Files read at that SHA:
  - `photonlib-java-examples/poseest/src/main/java/frc/robot/subsystems/drivetrain/SwerveDrive.java` (lines 165-174)
  - Cross-references the 9.1 note for the per-port enumeration.

## The verification

WPILib's `SwerveDrivePoseEstimator.addVisionMeasurement(Pose2d, double timestampSeconds, ...)`
documents the timestamp parameter as: "**The timestamp of the vision measurement in
seconds. Note that if you don't use your own time source by calling
SwerveDrivePoseEstimator.updateWithTime(double, Rotation2d, SwerveModulePosition[])
then you must use a timestamp with an epoch since FPGA startup (i.e., the epoch of this
timestamp is the same epoch as Timer.getFPGATimestamp())**."

So the contract is **explicit**: `timestampSeconds` must be on the FPGA epoch, matching
`Timer.getFPGATimestamp()`.

## Per-port verification

### Java
[SwerveDrive.java:165-174](photonlib-java-examples/poseest/src/main/java/frc/robot/subsystems/drivetrain/SwerveDrive.java#L165-L174):

    public void addVisionMeasurement(Pose2d visionMeasurement, double timestampSeconds) {
        poseEstimator.addVisionMeasurement(visionMeasurement, timestampSeconds);
    }

    public void addVisionMeasurement(
            Pose2d visionMeasurement, double timestampSeconds, Matrix<N3, N1> stdDevs) {
        poseEstimator.addVisionMeasurement(visionMeasurement, timestampSeconds, stdDevs);
    }

Direct passthrough of `timestampSeconds` to WPILib. Per 9.1 trace, the caller passes
`est.timestampSeconds` which Java's `getTimestampSeconds()` returns as
`metadata.captureTimestampMicros / 1e6`. On the robot, this is server-timebase µs ÷ 1e6
= FPGA-time seconds. **Java correctly satisfies WPILib's contract.** ✓

### C++
[Vision.h:97-98](photonlib-cpp-examples/poseest/src/main/include/Vision.h#L97-L98):

    estConsumer(visionEst->estimatedPose.ToPose2d(), visionEst->timestamp,
                GetEstimationStdDevs(visionEst->estimatedPose.ToPose2d()));

`visionEst->timestamp` is computed in `PhotonPipelineResult::GetTimestamp()` as
`ntReceiveTimestamp - GetLatency()`, where `ntReceiveTimestamp` was pre-set in
`GetAllUnreadResults` to `value.time - latency`. Net result: `value.time - 2*latency`,
the **5.2 BLOCKER**. C++ users pass this value directly to
`SwerveDrivePoseEstimator::AddVisionMeasurement` — which interprets it as FPGA-time
seconds.

So the C++ example **does treat the timestamp as FPGA-relative**, but the value it
passes is `T_capture - latency`, ~one frame in the past. WPILib trusts the FPGA-relative
interpretation; the result is a systematic ~16ms (at 60 fps) backward bias in the
vision-fused pose. ✗ (bug, not user error — already filed as 5.2 BLOCKER, escalated by
9.1).

### Python
[drivetrain.py:139-143](photonlib-python-examples/poseest/drivetrain.py#L139-L143):

    def addVisionPoseEstimate(self, pose: Pose3d, timestamp: float) -> None:
        self.poseEst.addVisionMeasurement(pose, timestamp)

Python passes `timestamp` directly. Python's `getTimestampSeconds()` returns
`(ntReceiveTimestampMicros - latency) / 1e6 = (T_pub - latency) / 1e6 = T_capture` (up to
network delay). FPGA-relative on the robot. **Python correctly satisfies WPILib's
contract** (modulo network delay of ~ms, much smaller than C++'s ~latency offset). ✓

## Cross-port table

| port   | Example call                                           | Timestamp produced       | Correct? |
| ------ | ------------------------------------------------------ | ------------------------ | -------- |
| Java   | `addVisionMeasurement(pose, est.timestampSeconds)`     | `T_capture`              | ✓        |
| C++    | `estConsumer(pose, visionEst->timestamp, stddev)`      | `T_capture - latency`    | ✗ (5.2 BLOCKER) |
| Python | `addVisionMeasurement(pose, timestamp)`                | `T_capture + network_delay` | ✓ (within ms)  |

All three ports **assume FPGA-relative**. Only C++ produces a value that doesn't satisfy
that assumption — and the bug isn't in the example (which is correct given its inputs);
it's in the library (the 5.2 BLOCKER).

## Outcome

**Clean for this check's primary question**: yes, all examples treat
`getTimestampSeconds()` as FPGA-relative, which matches WPILib's documented contract.
The 5.2 BLOCKER is the reason C++ produces a wrong value, not a flaw in the example
code itself.
