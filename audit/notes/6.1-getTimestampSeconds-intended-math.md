## Synthesis — what the docs collectively promise

The **collective doc intent** is:

> `getTimestampSeconds()` returns the frame-capture moment, expressed in the robot's clock
> (FPGA time, via wpi::nt::Now since the robot runs the TimeSync server). On the robot,
> server time == FPGA time. The math should reduce to "T_capture in robot's FPGA time".

The three ports differ in their internal recipe but **the documented result** is the
same: a robot-frame timestamp suitable to pass to `addVisionMeasurement`.

## The Python docstring is more honest

Python's docstring explicitly spells out the formula: `(NT Receive time - latency)`. The
Java/C++ docs are more abstract ("in the Time Sync Server's time base"), without saying
which clock is read and which terms are subtracted. This abstraction hides the
implementation divergence already filed at 4.2 + corrected at 5.2 BLOCKER.

## How the documented intent maps to a correct formula

To get "T_capture in robot's FPGA time", any of these are theoretically equivalent
(under perfect time-sync, no network delay):

| Formula                                  | Notes                                            |
| ---------------------------------------- | ------------------------------------------------ |
| (a) `captureTimestampMicros / 1e6`       | captureTimestampMicros is already server-base µs |
| (b) `(ntReceiveTimestamp - latency) / 1e6` | latency = publish - capture, both coproc-µs    |
| (c) `(now - latency) / 1e6` at receive moment | only correct if call is at the receive instant |

Java uses (a). Python uses (b). C++ uses (b) **but double-applies latency** (the
5.2 BLOCKER). C++'s deprecated GetLatestResult uses (c).

## Edge case: docs don't address what happens when

- `captureTimestampMicros = -1` (the default-constructor sentinel at
  [PhotonPipelineMetadata.java:48](photon-targeting/src/main/java/org/photonvision/targeting/PhotonPipelineMetadata.java#L48)).
  Java's `getTimestampSeconds()` returns `-1 / 1e6 = -1e-6 seconds`, a negative number
  that a robot interpreting as "FPGA time" sees as ~1 µs before FPGA boot. Probably
  rejected by the pose estimator's history buffer. NIT.
- `captureTimestampMicros = 0` from the 1.3 SMELL (VisionRunner disconnect path
  publishes `0 + offset`). Java returns `offset / 1e6` seconds — meaningful only as
  "server boot time relative to coproc boot time". Already filed at 1.3.
- Transient offset == 0 window (5.1 SMELL). Affects (a) and (b) differently — (a)
  publishes pre-corrected values that are wrong, (b)'s ntReceiveTimestamp is robot-time
  so the formula degrades more gracefully.

## Documentation gaps (NIT — not catalogued)

Across all three ports, the docs for `getTimestampSeconds`:
- Don't state the **return-on-error** convention (what if the result is uninitialized).
- Don't state monotonicity guarantees (or lack thereof).
- Don't mention behavior during time-sync transitions.
- Don't say "this can return a negative number" or "this can return Long.MIN_VALUE-equivalent
  seconds" for edge cases.

The docs imply "always returns a sane FPGA time" but the implementation has several
silent failure modes that escape that promise.

## Outcome

Doc-stated intent **across all three ports**: return `T_capture` in robot FPGA-time
seconds.

Implementation reality (from 4.2 + 5.2):
- Java: returns T_capture ≈ 1µs accuracy (time-sync error).
- Python: returns T_capture + network_delay ≈ ms accuracy.
- C++ (modern API `GetAllUnreadResults`): returns T_capture − latency ≈ tens of ms
  systematic offset. **BUG.**
- C++ (deprecated `GetLatestResult`): returns T_capture − network_delay (correct, like
  Python).

No new finding for 6.1 itself (this is a documentation-survey check). The pre-existing
4.2 DIVERGENCE + 5.2 BLOCKER + 5.1 SMELL collectively describe the implementation gaps.

Stage 6.2 should formally close by referencing this note + the prior findings.
