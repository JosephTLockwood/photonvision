## Three ports, three different formulas

### Java `getAllUnreadResults` ([PhotonCamera.java:253-268](photon-lib/src/main/java/org/photonvision/PhotonCamera.java#L253-L268))

    var changes = resultSubscriber.getAllChanges();
    for (var c : changes) {
        var result = c.value;
        // ignores c.timestamp entirely
        ret.add(result);
    }

Java's getAllUnreadResults intentionally **ignores** the NT4 receive timestamp
(`c.timestamp`). Comment at line 257-258: "We don't care about the timestamps from NT
- the metadata header has this, latency compensated by the Time Sync Client".

Downstream, Java `getTimestampSeconds()` returns `metadata.captureTimestampMicros / 1e6`
([PhotonPipelineResult.java:177-179](photon-targeting/src/main/java/org/photonvision/targeting/PhotonPipelineResult.java#L177-L179)).
On the robot (which runs the TimeSync server, so offset=0), this equals the coproc's
local capture-time mapped to robot's FPGA time. **One conversion. Accurate to time-sync
quality (~µs).**

### Python `getAllUnreadResults` ([photonCamera.py:139-152](photon-lib/py/photonlibpy/photonCamera.py#L139-L152))

    for change in changes:
        byteList = change.value
        timestamp = change.time
        ...
        newResult.ntReceiveTimestampMicros = timestamp   # NT4 send-time, single assignment

Python writes NT4's `change.time` directly into `ntReceiveTimestampMicros`. Then in
`getTimestampSeconds`
([photonPipelineResult.py:43-53](photon-lib/py/photonlibpy/targeting/photonPipelineResult.py#L43-L53)):

    latency = publishTimestampMicros - captureTimestampMicros
    return (self.ntReceiveTimestampMicros - latency) / 1e6

→ `(NT4_send_time - latency) / 1e6`. **One subtraction.** If NT4-send-time ≈ T_publish
on the robot's clock (after NT4 internal time-sync), this returns ≈ T_capture on robot
clock. Accurate to NT4 time-sync + network jitter (~ms).

### C++ `GetAllUnreadResults` ([PhotonCamera.cpp:212-249](photon-lib/src/main/native/cpp/photon/PhotonCamera.cpp#L212-L249))

    for (size_t i = 0; i < changes.size(); i++) {
        const wpi::nt::Timestamped<std::vector<uint8_t>>& value = changes[i];
        ...
        auto result = packet.Unpack<PhotonPipelineResult>();
        CheckTimeSyncOrWarn(result);
        // TODO: NT4 timestamps are still not to be trusted...
        result.SetReceiveTimestamp(wpi::units::microsecond_t(value.time) -
                                   result.GetLatency());                     // ← ONE subtraction here
        ret.push_back(result);
    }

C++ pre-subtracts latency from `value.time` before storing it in `ntReceiveTimestamp`.
Then `GetTimestamp()`
([PhotonPipelineResult.h:102-104](photon-targeting/src/main/native/include/photon/targeting/PhotonPipelineResult.h#L102-L104)):

    wpi::units::second_t GetTimestamp() const {
        return ntReceiveTimestamp - GetLatency();                            // ← ANOTHER subtraction
    }

→ `(value.time - latency) - latency = value.time - 2*latency`. **Two subtractions.**

C++ `GetLatestResult` (deprecated, but still present) at
[PhotonCamera.cpp:183-210](photon-lib/src/main/native/cpp/photon/PhotonCamera.cpp#L183-L210) does it
correctly:

    wpi::units::microsecond_t now = wpi::units::microsecond_t(
        wpi::RobotController::GetFPGATime());
    ...
    result.SetReceiveTimestamp(now);                                         // no pre-subtraction

→ `GetTimestamp() = now - latency`. Single subtraction. Approximates T_capture if called
right after receive.

## Numerical example

For a 60 fps camera with ~16 ms processing latency, ~2 ms network delay, calling
`getAllUnreadResults` on the robot:

| port   | `getTimestampSeconds()` result | error vs true T_capture |
| ------ | ------------------------------ | ----------------------- |
| Java   | `T_capture`                    | ~µs (time-sync error)   |
| Python | `T_capture + 2 ms`             | +network delay          |
| C++    | `T_capture − 14 ms`            | **−latency + network delay** |

C++ is roughly **one full latency in the past** compared to reality.

## Why this matters

- WPILib pose estimators (`SwerveDrivePoseEstimator`, etc.) match vision measurements
  against the odometry history buffer. The default buffer is ~1.5 s. A 14 ms error is
  well within the buffer, so the measurement is **accepted** with the wrong timestamp.
- Result: the C++ robot's vision-fused pose estimate is interpolated against odometry
  ~14 ms in the past, **systematically offset by one frame**.
- Symptom: vision-fused pose lags reality by ~one frame; corrections to fast-moving
  robot poses are mis-aligned.
- The error is consistent (not random), so it manifests as a smooth lag rather than
  jitter. Hard to notice without comparing to Java reference.

## Severity

**BLOCKER candidate.** The conditions:
- The error is systematic, not random.
- The recommended modern API (`GetAllUnreadResults`) is affected, while the deprecated
  one (`GetLatestResult`) is correct.
- Java and Python don't have this bug; this is a C++ regression.
- The magnitude is ~one frame at 60 fps — large enough to noticeably degrade a fast
  robot's pose estimate.
- WPILib's stale-rejection doesn't catch it (error << 1.5 s).

Conservative read: file as **BLOCKER** with `high` confidence. Math is unambiguous.

## Hand-derivation

Define on the robot's clock (FPGA time, equiv to NT4 server time since robot IS server):
- `T_cap` = coproc capture moment, mapped to robot FPGA time via the coproc's time-sync.
- `lat = publish_µs - capture_µs` (a duration on the coproc clock; equals the duration
  on the robot clock to time-sync precision).
- `T_pub = T_cap + lat`.
- `T_ntsend = T_pub` (NT4 publishes the value with the coproc's nt::Now timestamp; after
  time-sync conversion this is in robot's clock).
- `T_recv = T_pub + net_delay`.
- `value.time` on robot subscriber: NT4 sets this to the publisher-claimed publish-time,
  which is `T_pub` (before network delay), per NT4 spec. So `value.time = T_pub` (with
  some NT4 internal corrections).

The two C++ steps:
1. `ntReceiveTimestamp = value.time - lat = T_pub - lat = T_cap`.
2. `GetTimestamp() = ntReceiveTimestamp - lat = T_cap - lat`. ← off by one latency.

Whereas Python's single step:
1. `ntReceiveTimestamp = value.time = T_pub`.
2. `getTimestampSeconds = (ntReceiveTimestamp - lat) / 1e6 = (T_pub - lat) / 1e6 = T_cap`. ✓

So the bug is **exactly one extra latency subtraction** in the C++ path.

## What about the Java getTimestampSeconds DIVERGENCE I filed at 4.2?

The 4.2 DIVERGENCE said "Java uses absolute capture time; C++/Python use receive minus
latency". I treated those as semantically equivalent under perfect time-sync. They're
not — Java is correct, Python is correct (up to network delay), C++ is wrong by an extra
latency.

The 4.2 finding's "minor numerical difference" framing **understated** the C++ bug. The
correction is: Java + Python both produce ≈ T_capture (within time-sync error or network
delay); C++ produces T_capture − latency (systematic offset of one full processing
latency).
