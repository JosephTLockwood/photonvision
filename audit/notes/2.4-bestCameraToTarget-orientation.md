## The name's claim

`PhotonTrackedTarget.java:54-58`:

    /** The transform with the lowest reprojection error */
    public Transform3d bestCameraToTarget;
    /** The transform with the highest reprojection error */
    public Transform3d altCameraToTarget;

The field name reads "camera → target" (the target's pose expressed in the camera's
frame). FRC/WPILib convention for `Transform3d` named `aToB` is: applying it to a
`Pose3d` at frame A gives the pose of B in the same reference frame. So
`bestCameraToTarget` applied to camera pose = target pose. Equivalently: target's pose
in camera frame.

## Trace 1 — AprilTag detector path (coprocessor)

WPILib `AprilTagPoseEstimate.pose1/pose2`: returned by the AprilTag detector. Per
`MathUtils.java:191`:

> "our detected transformation will be in EDN. Our subsequent uses of the transformation,
> however, assume the tag's z-axis points away from the tag instead of into it."

So:
1. WPILib detector → `Transform3d` in **EDN**, tag-Z pointing into tag (away from camera).
2. `TrackedTarget` ctor at
   [TrackedTarget.java:113-117](photon-core/src/main/java/org/photonvision/vision/target/TrackedTarget.java#L113-L117):
   applies `MathUtils.convertApriltagtoOpenCV(bestPose)` → rotates tag-frame's rotation
   by 180° around its Y axis (`APRILTAG_BASE_ROTATION` at
   [MathUtils.java:195-196](photon-core/src/main/java/org/photonvision/common/util/math/MathUtils.java#L195-L196)).
   Result: still in **EDN**, tag-Z now points out of tag (toward camera). Stored in
   `m_bestCameraToTarget3d`.
3. `AprilTagPipeline.java:226-234` then applies `MathUtils.convertOpenCVtoPhotonTransform`
   (EDN → NWU). Stores the NWU version back via `setBestCameraToTarget3d`.
4. `TrackedTarget.simpleFromTrackedTargets`
   ([TrackedTarget.java:455](photon-core/src/main/java/org/photonvision/vision/target/TrackedTarget.java#L455))
   propagates this to `PhotonTrackedTarget.bestCameraToTarget`.

So the on-the-wire `PhotonTrackedTarget.bestCameraToTarget` is in **NWU coordinates,
camera frame on the left, tag frame on the right with tag-Z out of face**.

## Trace 2 — Multi-tag path (within AprilTagPipeline)

When per-tag pose is missing but multi-tag result is available, PV synthesizes per-tag poses:
[AprilTagPipeline.java:200-213](photon-core/src/main/java/org/photonvision/vision/pipeline/AprilTagPipeline.java#L200-L213):

    var camToTag = new Transform3d(new Pose3d().plus(multiTagResult.get().estimatedPose.best), tagPose.get());
    camToTag = CoordinateSystem.convert(camToTag, NWU, EDN);     // NWU → EDN
    camToTag = new Transform3d(camToTag.getTranslation(),
                               new Rotation3d(0, π, 0).rotateBy(camToTag.getRotation()));
                                                                  // tag-Z out → tag-Z into tag
    tagPoseEstimate = new AprilTagPoseEstimate(camToTag, camToTag, 0, 0);

This **manually undoes** the (NWU + tag-Z-out) → (EDN + tag-Z-into) conversion so that
the synthesized `AprilTagPoseEstimate` looks like what the AprilTag detector would have
returned. The TrackedTarget ctor then re-applies the same two conversions back. Round-trip
preserves the data. ✓

## Trace 3 — SolvePNPPipe (non-fiducial 3D solvePnP)

[SolvePNPPipe.java:100-103](photon-core/src/main/java/org/photonvision/vision/pipe/impl/SolvePNPPipe.java#L100-L103):

    Transform3d camToTarget = MathUtils.convertOpenCVtoPhotonTransform(new Transform3d(translation, rotation));
    target.setBestCameraToTarget3d(camToTarget);
    target.setAltCameraToTarget3d(new Transform3d());

`translation`/`rotation` are read directly from OpenCV's `solvePnP` rvec/tvec (EDN).
Single `convertOpenCVtoPhotonTransform` (EDN → NWU) yields NWU. ✓

NIT: `altCameraToTarget3d` is set to default `new Transform3d()` (identity). For non-fiducial
3D solvePnP there's no alternate solution from IPPE; identity is a benign placeholder
because reprojection-error / ambiguity also default to 0, so consumers can't
mis-interpret it as a meaningful alternate.

## Trace 4 — OpenCVHelp.solvePNP_SQUARE (used by photon-lib / sim, not coprocessor)

[OpenCVHelp.java:560-565](photon-targeting/src/main/java/org/photonvision/estimation/OpenCVHelp.java#L560-L565):
`best = new Transform3d(tvecToTranslation(...), rvecToRotation(...))`. Both helpers
internally call `translationEDNtoNWU` / `rotationEDNtoNWU`. The result is **NWU camera
frame**, tag's local frame coming straight from OpenCV's interpretation of `tagModel.vertices`.

`tagModel.vertices` is documented at
[OpenCVHelp.java:502-509](photon-targeting/src/main/java/org/photonvision/estimation/OpenCVHelp.java#L502-L509) in NWU with all points at X=0
(tag plane = YZ). So the tag's local frame has X-out-of-face. This matches WPILib
`AprilTagFieldLayout`'s tag-pose convention (also X-out-of-face). ✓

## Consumer-side use

`PhotonPoseEstimator.LOWEST_AMBIGUITY` at
[PhotonPoseEstimator.java:498-504](photon-lib/src/main/java/org/photonvision/PhotonPoseEstimator.java#L498-L504):

    return Optional.of(
        new EstimatedRobotPose(
            fieldTags.getTagPose(lowestAmbiguityTarget.getFiducialId()).get()
                .transformBy(lowestAmbiguityTarget.getBestCameraToTarget().inverse())
                .transformBy(robotToCamera.inverse()),
            ...));

- `fieldTags.getTagPose(id)` = tag's pose in **NWU field frame**, tag's local frame **X-out**.
- `bestCameraToTarget.inverse()` = `T_tag_to_camera` (camera's pose in tag's local frame).
- Applying that to the field-frame tag pose gives camera's pose in field — correct math
  if both sides agree on the tag's local frame.

PV's tag local frame (X-out) and WPILib's `AprilTagFieldLayout` tag local frame (X-out)
agree. ✓

## Cross-port

C++ has matching `VisionEstimation::EstimateCamPosePNP`, `MathUtils::ConvertApriltagToOpenCV`,
and pipeline-level EDN→NWU conversions. Python's `photonlibpy` only consumes
`bestCameraToTarget` from the wire; no production-side conversion needed. All three ports
expect the on-the-wire transform to be NWU with tag-Z out of face.

## Outcome

**Clean.** The naming `bestCameraToTarget` (camera-frame on left, tag-frame on right)
matches every code path's actual behavior at the wire level. Conversions are deliberate
and consistent, just split across multiple files for historical reasons.
