## Call site A — `OpenCVHelp.solvePNP_SQUARE` (the canonical AprilTag PnP path)

Inputs:
- `modelTrls` — 3D corner translations in **WPILib NWU** marker frame, documented order
  (lines 502-509):
    Point 0: `[0, -L/2,  L/2]`   (X=0, Y=-L/2, Z=+L/2)  → TR in image when facing camera
    Point 1: `[0,  L/2,  L/2]`                          → TL
    Point 2: `[0,  L/2, -L/2]`                          → BL
    Point 3: `[0, -L/2, -L/2]`                          → BR
  i.e., `[TR, TL, BL, BR]` (CW starting from image-TR).
- `imagePoints` — pixel positions in matching order.

The function calls `reorderCircular(elements, backwards=true, shiftStart=-1)` on **both**
modelTrls and imagePoints (lines 533-534) before passing to OpenCV. Hand-tracing
`reorderCircular([A,B,C,D], true, -1)`:

    size=4, dir=-1
    i=0:  index = (0*-1 + -1*-1) % 4 = 1     → out[0] = elements[1] = B
    i=1:  index = (1*-1 + -1*-1) % 4 = 0     → out[1] = elements[0] = A
    i=2:  index = (2*-1 + -1*-1) % 4 = -1+4=3 → out[2] = elements[3] = D
    i=3:  index = (3*-1 + -1*-1) % 4 = -2+4=2 → out[3] = elements[2] = C

So `reorderCircular(_, true, -1)` is the swap-pairs permutation `(0↔1, 2↔3)`. It is its
own inverse.

Applied to documented modelTrls `[TR, TL, BL, BR]` → `[TL, TR, BR, BL]`. That **matches
OpenCV's `SOLVEPNP_IPPE_SQUARE` expected order** (OpenCV docs):
- Point 0 = TL = `(-L/2, +L/2, 0)`
- Point 1 = TR = `(+L/2, +L/2, 0)`
- Point 2 = BR = `(+L/2, -L/2, 0)`
- Point 3 = BL = `(-L/2, -L/2, 0)`

(The NWU→EDN axis conversion via `translationToTvec` is a separate per-point coordinate
transform; it preserves the **ordering** of points, so the index correspondence between
post-reorder modelTrls and post-reorder imagePoints is preserved.)

**Same reorder is applied to both 3D and 2D inputs**, so the index pairing
`objectPoints[i] ↔ imagePoints[i]` is preserved through the reordering. **No
correspondence breakage.**

Verdict: **OpenCVHelp.solvePNP_SQUARE looks consistent.** Has been in production for
multiple seasons.

## Call site B — `ArucoPoseEstimatorPipe.process` (the legacy ArUco path)

This is a separate code path used when ArUco detection (vs the wpilib AprilTag detector)
is the active detector.

**Object points** (lines 128-132, set once when `tagSize` is set):

    objectPoints[0] = (-L/2, +L/2, 0)  → TL in OpenCV marker frame
    objectPoints[1] = (+L/2, +L/2, 0)  → TR
    objectPoints[2] = (+L/2, -L/2, 0)  → BR
    objectPoints[3] = (-L/2, -L/2, 0)  → BL

This is **exactly OpenCV's `SOLVEPNP_IPPE_SQUARE` expected order** `[TL, TR, BR, BL]`. ✓

**Image points** (lines 75-80, populated each frame):

    // We receive 2d corners as (BL, BR, TR, TL) but we want (BR, BL, TL, TR)
    imagePoints[0] = in[1]   (if comment is right: BR)
    imagePoints[1] = in[0]   (BL)
    imagePoints[2] = in[3]   (TL)
    imagePoints[3] = in[2]   (TR)

So the comment claims final order = `[BR, BL, TL, TR]`.

For solvePnP to be correct, `imagePoints[i]` must be the image projection of
`objectPoints[i]` for the same physical corner. Under face-on geometry:

- `objectPoints[0]` = TL_marker. With +X right and +Y up in marker frame, and image's +Y
  pointing down, marker TL projects to image-TL.
- So **correct** `imagePoints[0]` for face-on tag = image-TL.

But the code sets `imagePoints[0] = BR` (per the comment). That is **OpenCV's order
`[TL, TR, BR, BL]` cyclically rotated by 2 (i.e., 180°)**. If the comment about the
incoming order is accurate, this corresponds to feeding solvePnP the marker rotated
180° around its Z axis.

### Tracing the upstream comment chain

`PhotonArucoDetector.detect()` lines 93-106 (cited verbatim above):

    // ArUco detection returns corners (BR, BL, TL, TR).
    // For parity with AprilTags and photonlib, we want (BL, BR, TR, TL).
    double[] xCorners = { cornerMat.get(0, 1)[0], cornerMat.get(0, 0)[0],
                          cornerMat.get(0, 3)[0], cornerMat.get(0, 2)[0] };

This is the same swap-pairs permutation. Per the chain of comments:

- Raw OpenCV ArUco: claimed `[BR, BL, TL, TR]`.
- After PhotonArucoDetector swap-pair: `[BL, BR, TR, TL]` (matches second comment).
- After ArucoPoseEstimatorPipe swap-pair: `[BR, BL, TL, TR]` (the original raw order!).

The two swaps cancel out. So the imagePoints passed to solvePnP are **whatever raw order
OpenCV ArUco returned, unchanged**.

### The unverified premise

The whole correctness argument hinges on: **does OpenCV ArUco's
`detectMarkers` actually return corners in `[BR, BL, TL, TR]` order**, as the PV comment
asserts?

OpenCV's official documentation
([cv::aruco::detectMarkers](https://docs.opencv.org/4.x/d9/d6a/group__aruco.html))
states the corners are returned **clockwise from the top-left** of the marker, i.e.
`[TL, TR, BR, BL]` in image-pixel positions when the tag faces the camera with no
rotation. This **conflicts** with the PV comment.

If the OpenCV docs are correct and the PV comment is wrong:
- Raw `[TL, TR, BR, BL]` (OpenCV)
- After PhotonArucoDetector swap-pair: `[TR, TL, BL, BR]`
- After ArucoPoseEstimatorPipe swap-pair: `[TL, TR, BR, BL]`
- imagePoints[i] now correctly corresponds to objectPoints[i]. Path **would be correct**.

If the OpenCV docs are wrong and the PV comment is right:
- imagePoints[i] would be the corner 180° around from objectPoints[i].
- The recovered pose would be the tag rotated 180° around its Z axis.

I cannot resolve this conflict from in-tree code alone. The `detector` instance at
`PhotonArucoDetector.java:86` is OpenCV's `cv::aruco::ArucoDetector`, whose
`detectMarkers` semantics live in OpenCV's source.

### Hand-derivation of which corner OpenCV ArUco calls "corner 0"

OpenCV's contour-extraction in ArUco scans the image and reports the first corner of the
quadrilateral after canonicalizing rotation by the marker's internal code. The "TL" in
their docs is the **marker's logical top-left** (determined by the bits inside the
marker), not the image's geometric top-left. So if a marker is physically rotated 180°,
OpenCV will still call its "TL" corner index 0 — and that corner lands in the image
at the bottom-right of the marker.

This means the PV comment "(BR, BL, TL, TR)" could be describing a specific test
condition (markers rotated 180° from a reference orientation), or could be reading the
marker's internal-orientation labels as image positions, or could be a stale comment from
before the OpenCV ArUco API stabilized.

## Severity assessment

- **OpenCVHelp.solvePNP_SQUARE** (the main AprilTag PnP path): the code is
  index-self-consistent, since the same reordering is applied to 3D and 2D inputs.
  Documented input contract is clear. **No finding here.**
- **ArucoPoseEstimatorPipe** (the legacy ArUco path): correctness depends on a comment
  claim about OpenCV's return order that conflicts with OpenCV's published docs. The two
  swap-pair reorders cancel, so the imagePoints fed to solvePnP are whatever OpenCV
  returned natively. If OpenCV returns `[TL, TR, BR, BL]` (per docs), the pairing is
  correct. If OpenCV returns `[BR, BL, TL, TR]` (per PV comment), the recovered pose is
  rotated 180° around the tag's normal.

follow-up sub-check `2.1a` to verify with a synthetic run.

## Adjacent observations

- The chain `PhotonArucoDetector` → `ArucoDetectionResult` → `ArucoPoseEstimatorPipe`
  involves two swap-pair reorderings that **cancel**. Whatever motivated the parity
  comment "For parity with AprilTags and photonlib, we want (BL, BR, TR, TL)" — the
  cancellation defeats that purpose for the IPPE PnP step. If the goal was to deliver
  AprilTag-ordered corners to the PnP, the second reorder undoes it. SMELL but harmless
  if (as the comments imply) it's just shuffling for nothing.
- The comment in `OpenCVHelp.solvePNP_SQUARE` lines 502-509 documents the **WPILib NWU**
  3D point order. There is no corresponding documentation of the 2D `imagePoints`
  expected order — the Javadoc just says "The order should match the given object point
  translations." Anyone calling this function needs to know the 3D order in their head
  AND apply the same image-corner order. Light doc gap.
