## What gets reported

`OpenCVHelp.solvePNP_SQPNP` returns `PnpResult(best, error[0])` (2-arg ctor)
([OpenCVHelp.java:673](photon-targeting/src/main/java/org/photonvision/estimation/OpenCVHelp.java#L673)).
The 2-arg ctor at
[PnpResult.java:68-70](photon-targeting/src/main/java/org/photonvision/targeting/PnpResult.java#L68-L70)
expands to `PnpResult(best, best, 0, bestReprojErr, bestReprojErr)`:

| field         | value                           |
| ------------- | ------------------------------- |
| `best`        | OpenCV solution (single)        |
| `alt`         | = `best` (placeholder)          |
| `ambiguity`   | `0.0`                           |
| `bestReprojErr` | OpenCV `error[0]` (RMS pixels) |
| `altReprojErr`  | = `bestReprojErr`              |

C++ at
[OpenCVHelp.h:278+](photon-targeting/src/main/native/include/photon/estimation/OpenCVHelp.h#L278)
mirrors this. Python at
[openCVHelp.py:312-313](photon-lib/py/photonlibpy/estimation/openCVHelp.py#L312-L313):
`PnpResult(best=best, bestReprojErr=error, alt=best, altReprojErr=error)` â€” `ambiguity`

All three ports: **ambiguity always = 0 for multi-tag.**

## Is this stubbed or genuine?

Genuine, not stubbed. OpenCV's `SOLVEPNP_SQPNP` is a **single-solution** PnP solver by
design (per OpenCV docs: "It returns just one solution."). There is no alternate pose
to ratio against. Reporting `ambiguity = 0` exactly matches the documented contract at
[PnpResult.java:56](photon-targeting/src/main/java/org/photonvision/targeting/PnpResult.java#L56):
"If no alternate solution is found, this is 0".

Contrast with single-tag IPPE_SQUARE which **does** return two solutions and computes
`ambiguity = errors[0] / errors[1]`.

## Consumer side

`PoseStrategy.MULTI_TAG_PNP_ON_COPROCESSOR` and `MULTI_TAG_PNP_ON_RIO`
([PhotonPoseEstimator.java:413-466](photon-lib/src/main/java/org/photonvision/PhotonPoseEstimator.java#L413-L466))
do **not** consult `ambiguity` at all. They take `multitagResult.get().estimatedPose.best`
and apply it directly. So a consumer wanting quality-gating for multi-tag results must
read `bestReprojErr` explicitly â€” there's no ambiguity-based gate.

`PoseStrategy.AVERAGE_BEST_TARGETS` consults each individual target's per-tag
`getPoseAmbiguity()`, not the multi-tag combined ambiguity.

## Adjacent gap: per-target reprojection error in multi-tag context

The multi-tag `bestReprojErr` is the **aggregate** RMS reprojection error across **all
tag corners** (~4Ă—N values for N visible tags). A single outlier tag with bad detection
contaminates this single scalar. Per-tag reprojection errors are not surfaced â€” there's
no field for that in `MultiTargetPNPResult`. A consumer can't tell which tag is hurting
the fit.

This is a structural gap rather than a bug: the wire schema has no slot for per-tag
errors. Recording, not cataloging.

## Cross-port consistency

- Java/C++/Python all use OpenCV `SOLVEPNP_SQPNP` (or its Python `cv.SOLVEPNP_SQPNP` flag).
- All three default `ambiguity = 0` and reuse the single error scalar for both `best/alt
  ReprojErr`.
- No port divergence.

## Verdict

**Clean.** Multi-tag ambiguity = 0 is semantically correct given the single-solution
solver. Reprojection error is correctly populated (per check 2.3, units are pixels).

Adjacent doc gap: `PnpResult` Javadoc doesn't note that **multi-tag SQPnP always
reports ambiguity = 0** (and ambiguity is therefore not a useful quality signal for
the MULTI_TAG_PNP_* strategies). Worth a doc tweak, not a finding.
