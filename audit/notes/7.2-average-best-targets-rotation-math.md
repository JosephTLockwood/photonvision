## Translation is fine

`transform.plus(translation.times(weight))` accumulates a weighted sum of translations.
This **is** a proper weighted average of translations (since translations are in a flat
vector space). ✓

## Rotation is mathematically incorrect

`rotation.rotateBy(R_i.times(weight_i))` accumulates a **non-commutative quaternion
product**:

    rotation_after_N = q_N^{w_N} · q_{N-1}^{w_{N-1}} · ... · q_1^{w_1}

where `q_i` is the i-th target's estimated rotation and `w_i` is its weight. (Recall
WPILib's `rotateBy(other)` computes `other.q * self.q` — left-multiplication.)

`R.times(w)` semantics: scale the angle of R by w while preserving its axis. Equivalent
to `Quaternion.exp(w * Quaternion.log(R.getQuaternion()))`. So `q^w` denotes the
scaled-angle quaternion.

The "proper" weighted average of rotations would be one of:
- **Quaternion average via Markley's eigenvector method**: closed-form for arbitrary weights.
- **Tangent-space average**: `Quaternion.exp(sum(w_i * Quaternion.log(q_i)))`. Approximate
  but order-independent.

Neither equals the iterative product. They diverge when the q_i are far apart in rotation
space.

## Hand-derivation showing order-dependence

Take 2 targets with equal weights (w_1 = w_2 = 0.5):
- Target 1: q_1 = rotation 90° about Z (yaw 90°).
- Target 2: q_2 = rotation 90° about X (roll 90°).

PV's iterative product (input order 1, 2):

    rotation_0 = I
    rotation_1 = q_1^{0.5} · I            = 45° about Z
    rotation_2 = q_2^{0.5} · rotation_1   = (45° about X) · (45° about Z)

Now reverse input order (target 2 first, then target 1):

    rotation_0 = I
    rotation_1 = q_2^{0.5} · I            = 45° about X
    rotation_2 = q_1^{0.5} · rotation_1   = (45° about Z) · (45° about X)

These two final rotations are **different** (quaternion multiplication isn't
commutative): (45°X)·(45°Z) ≠ (45°Z)·(45°X). A proper weighted average of two equal-weight
rotations should be the **midpoint** along the geodesic between q_1 and q_2 — clearly
order-independent.

So PV's `AVERAGE_BEST_TARGETS` answer **depends on the iteration order of `targets`**.
In `cameraResult.targets`, the order is determined by PhotonVision's sort mode
(typically largest target first, but configurable). Frame-to-frame target reordering
shifts the computed pose even when individual per-target estimates are stable.

## When does the math give an OK answer?

**Case A — all rotations equal** (q_i ≡ q): product = `q^{sum w_i} = q^1 = q`. Since the
weights sum to 1 by construction
([PhotonPoseEstimator.java:736](photon-lib/src/main/java/org/photonvision/PhotonPoseEstimator.java#L736)
divides by `totalAmbiguity` = `sum(1/amb_i)`), this case gives the **correct** rotation.
This is the typical multi-tag scenario (all tags estimate the same robot pose).

**Case B — small rotations close to identity** (q_i ≈ I): by Baker-Campbell-Hausdorff
expansion, `exp(A) · exp(B) ≈ exp(A + B) + (1/2)[A, B] + ...`. The commutator [A, B] is
small when A and B are small, so the product is **approximately** the tangent-space sum.
First-order correct.

**Case C — large disparate rotations**: BCH commutator is significant. Product is
**order-dependent** and not a proper geodesic mean.

For multi-tag detection from a single camera, the per-target rotation estimates are
within a few degrees of each other (they're all estimating the same robot pose, just
with different per-tag noise sources). Case A/B typically applies; the resulting error
is sub-degree.

**But:** if the targets are far apart in the FOV with different distortion characteristics,
or one tag has a very bad PnP fit while others are good, the rotation disagreement can
exceed several degrees. In that case the iterative product produces a noticeably wrong
"average", and the wrongness is order-dependent.

## Severity

**SMELL.** The math is provably approximate (not the documented "weighted average"); the
real-world error is typically sub-degree for healthy multi-tag scenarios; the
order-dependence is a real source of frame-to-frame jitter that's hard to attribute.

The audit prompt explicitly called this out as suspicious — confirmed.

Filing as SMELL with **high** confidence (math is unambiguous, hand-derivation
demonstrates the order-dependence).
