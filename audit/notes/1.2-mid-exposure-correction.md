## Coprocessor / capture side (Java only — no C++/Python coprocessor code)

PhotonVision's coprocessor is Java-only (no `.cpp` / `.py` under `photon-core/`). All capture
paths produce `Frame.timestampNanos` without any exposure-duration correction:

- **USBFrameProvider** ([USBFrameProvider.java:71-75, :96-100](photon-core/src/main/java/org/photonvision/vision/frame/provider/USBFrameProvider.java#L71-L100))
  — uses `cvSink.grabFrame(...)` which returns the cscore frame time, which (per the design
  doc) is the V4L2 buffer timestamp on Linux. No exposure offset applied.
- **LibcameraGpuFrameProvider** ([LibcameraGpuFrameProvider.java:86-100](photon-core/src/main/java/org/photonvision/vision/frame/provider/LibcameraGpuFrameProvider.java#L86-L100))
  — uses `wpiNanoTime() - (libcameraNow - libcameraCapture)`. The libcamera "capture"
  timestamp is `SensorTimestamp` (typically end-of-readout, not start, not mid). No
  exposure offset applied.

## Sim side

- **Java sim** ([VisionSystemSim.java:424-426](photon-lib/src/main/java/org/photonvision/simulation/VisionSystemSim.java#L424-L426)):
  `timestampCapture = timestampNT / 1e6 - latencyMillis / 1e3`. `latencyMillis` is the
  configured processing latency only; `exposureTimeMs` is **only** used to clamp `frameSpeedMs`
  (see [SimCameraProperties.java:241,255](photon-lib/src/main/java/org/photonvision/simulation/SimCameraProperties.java#L241)).
  → sim publishes `capture = publish − processing_latency`. Anchor = "instant the
  frame existed", i.e. consistent with **start of exposure** under the contract.
- **Python sim** ([photonCameraSim.py:441](photon-lib/py/photonlibpy/simulation/photonCameraSim.py#L441)):
  `captureTimestampMicros = int(publishTimestampMicros - latency * 1e6)`. Identical
  semantics to Java sim.

## The contract (design doc, the public authority)

`docs/source/docs/contributing/design-descriptions/e2e-latency.md` at lines 32-38 is the
closest thing to a written contract:

- Lines 33-34: "On cameras that support `V4L2_BUF_FLAG_TSTAMP_SRC_SOE`, the V4L2 buffer
  timestamp = Start Of Exposure."
- Line 38 (verbatim): *"This gets us an estimate at when the camera **started** collecting
  photons. The camera's sensor will remain collecting light for up to the total integration
  time, plus readout time for rolling shutter cameras."*

So the doc:
- Acknowledges the timestamp is start-of-exposure (best case) or worse (rolling-shutter or
  non-SOE-capable cameras).
- Acknowledges the residual error (exposure time + rolling-shutter smear) but does **not**
  promise mid-exposure correction.

## What's actually missing

1. **Mid-exposure correction is not implemented anywhere.** This matches
   `memory/audit_prefound.md` → PR #2489 + driver #33 are the in-flight fix. **Not
   re-cataloging as a finding.**
2. **The public Javadoc on `PhotonPipelineMetadata.getCaptureTimestampMicros()`
   ([PhotonPipelineMetadata.java:61-65](photon-targeting/src/main/java/org/photonvision/targeting/PhotonPipelineMetadata.java#L61-L65))
   says only "The time that this image was captured". No mention of:
   - which anchor (SOE/EOF/mid),
   - that the value can vary by which V4L2 flag the camera advertises,
   - that rolling-shutter cameras smear illumination across the readout window.
   This is a doc-completeness gap, not a code bug. Listed below as a secondary thread,
   not catalogued as a separate finding (overlaps with the 1.1 finding which already calls
   out the same getter's doc problem). Worth folding into PR #2489's doc update if not
   already covered.
3. **`V4L2_BUF_FLAG_TSTAMP_SRC_*` per-buffer flag is not exposed by cscore.** PhotonVision
   has no way to detect at runtime whether the current frame's timestamp is SOE or EOF —
   the design doc just trusts that the configured cameras report SOE. A camera silently
   demoted to EOF mode (e.g. firmware update, kernel driver change) would shift the
   timestamp anchor by one frame-period and no consumer would notice. This is upstream
   (cscore) and out-of-scope to fix here, but worth noting as a system-level fragility.
   Not cataloged.
