## What's already on the wire per `PhotonPipelineResult`

For each `PhotonTrackedTarget`:
- `poseAmbiguity` (float64, see check 2.2): best/alt reprojection-error ratio,
  range `[0, 1]` under correct conditions, lower is more confident. NaN/Inf escapes
  possible per the 2.2 SMELL.
- `bestCameraToTarget.getTranslation().getNorm()` (float64, meters): Euclidean
  distance from camera to tag. Per the standard PnP error model, position uncertainty
  scales roughly as `distance^2 / focal_length` (linear in distance for small angles,
  growing for tags farther away).
- `bestCameraToTarget.getRotation()` (Rotation3d): tilt of the tag. Tags viewed
  off-axis give worse PnP fits than head-on tags (the IPPE solver becomes ambiguous
  at low incidence angles).
- `fiducialId` (int): which tag was used; can index into per-tag calibration history
  if maintained externally.
- `minAreaRectCorners.area()` (derived): image-pixel-area of the detection. Larger
  detections = more pixels involved = better PnP precision.

For each `MultiTargetPNPResult` (when present):
- `estimatedPose.bestReprojErr` (float64, pixels — see check 2.3): RMS reprojection
  error of the multi-tag SQPnP fit. Aggregate quality of the whole set.
- `estimatedPose.altReprojErr` (float64): same as bestReprojErr for single-solution
  SQPnP (per 3.3 clean result).
- `fiducialIDsUsed.size()` (int): number of tags contributing to the fit. More tags
  = better-constrained.

## Mapping to a stddev for `addVisionMeasurement`

The robot-side stddev that `SwerveDrivePoseEstimator.addVisionMeasurement` consumes
is a `Matrix<N3, N1>` for `[σ_x, σ_y, σ_theta]` in meters and radians.

A typical heuristic robot author can construct:

    σ_xy = base_xy * (1 + k_dist * distance² + k_amb * ambiguity + k_repr * reprojErr)
    σ_θ  = base_θ  * (1 + k_dist * distance² + k_amb * ambiguity + k_repr * reprojErr)

Where `base_*` are tuned constants and `k_*` are scaling weights. Distance² is the
most-load-bearing term (PnP error grows quadratically with distance for small angles).

## Why PhotonVision can't trivially compute this internally

- It doesn't know the **robot's noise floor** — sensor read latency, drivetrain
  vibration, encoder noise all factor in.
- It doesn't know **which downstream filter** the user is feeding (Kalman gain
  semantics differ across filter implementations).
- Different consumers want different tradeoffs (e.g., low-stddev for "trust vision
  more during teleop", high-stddev for "ignore vision during fast spins").

So the deliberate choice to leave stddev to the consumer is reasonable, but it
means PV could surface its **quality signals** more prominently:

## What's missing structurally

1. **No aggregate quality field on `EstimatedRobotPose`**. Per check 8.1, the dataclass
   carries `(pose, timestamp, targets, [strategy])` — no quality scalar.
2. **No per-tag reprojection error on `PhotonTrackedTarget`**. The detector knows the
   PnP fit residual per tag, but only the multi-tag `bestReprojErr` (aggregate) is
   surfaced via `MultiTargetPNPResult.estimatedPose`.
3. **CONSTRAINED_SOLVEPNP discards its residual** (filed as 2.3 SMELL).
4. **AVERAGE_BEST_TARGETS doesn't report its own internal stddev** — but it computes
   weighted per-pose residuals that could feed a stddev (the weights themselves are
   inverse-ambiguity).

## Adjacent suggestions (not findings)

If a future API change adds a `standardDeviations: Matrix<N3, N1>` field to
`EstimatedRobotPose`, the population logic could be:

| PoseStrategy            | Recommended stddev source                                   |
| ----------------------- | ----------------------------------------------------------- |
| LOWEST_AMBIGUITY        | Scale base stddev by best target's `(distance², ambiguity)` |
| CLOSEST_TO_CAMERA_HEIGHT | Same as LOWEST_AMBIGUITY                                   |
| CLOSEST_TO_REFERENCE_POSE | Same as LOWEST_AMBIGUITY                                  |
| AVERAGE_BEST_TARGETS    | Inverse of total ambiguity (proxy for confidence)           |
| MULTI_TAG_PNP_*         | Scale base stddev by `(bestReprojErr, 1/sqrt(N_tags))`      |
| PNP_DISTANCE_TRIG_SOLVE | Scale base stddev by `distance²` (single tag, no multi-tag) |
| CONSTRAINED_SOLVEPNP    | Final solver residual (which is currently discarded — 2.3)  |

These are recipes; PhotonVision could surface them as `getStddev()` methods on
`EstimatedRobotPose` and let robot code apply per-robot tuning constants on top.
