## Implication for downstream WPILib pose estimators

WPILib's `SwerveDrivePoseEstimator.addVisionMeasurement(Pose2d, double timestamp,
Matrix<N3, N1> visionMeasurementStdDevs)` requires a 3-element stddev vector
(x, y, heading). Photon's `EstimatedRobotPose` doesn't carry one, so robot code must
supply it from external knowledge. Common patterns observed in FRC code:

1. **Hardcoded constants** — e.g., `VecBuilder.fill(0.5, 0.5, 0.5)`. Robust but ignores
   actual measurement quality.
2. **Distance-scaled heuristic** — increase stddev linearly with average tag distance.
3. **Ambiguity-scaled heuristic** — multiply stddev by `target.getPoseAmbiguity()`.
4. **No stddev** — use the default which WPILib internally treats as ~0.1m/rad.

None of these reflect the actual PnP reprojection-error magnitude that PhotonVision
already computes internally (visible via `PnpResult.bestReprojErr`, see check 2.3).

## Candidate signals already on the wire (for Stage 8.2)

The `PhotonPipelineResult` and its embedded targets already carry the following
quality signals that *could* feed a stddev:

- `target.poseAmbiguity` — best/alt reprojection-error ratio (0..1, lower is more confident).
- `target.bestCameraToTarget.getTranslation().getNorm()` — distance to tag (uncertainty
  scales roughly with tag-distance squared in PnP).
- `multitagResult.estimatedPose.bestReprojErr` — RMS pixel error of the multi-tag fit.
- (Implicit) number of tags used — more tags = better constrained.

None of these are surfaced as a stddev field on `EstimatedRobotPose`.

## Is this a finding?

The audit prompt explicitly asks to "document the gap", not to file a high-severity
finding. The absence is **deliberate** (PV's design has always required the consumer
to supply stddev externally). Filing as a SMELL would be a stretch since:
- There's no contract that promises a stddev.
- The "alternative" (computing stddev internally) requires knowledge of the robot's
  noise model that PV doesn't have.
- A typical robot codebase wires up a hardcoded stddev that works fine in practice.

**Decision: do NOT file a separate finding.** The note itself is the deliverable. The
2.3 SMELL (CONSTRAINED_SOLVEPNP discards residual) already touches the related concern
that PnP quality signal is sometimes truncated even where it exists.
