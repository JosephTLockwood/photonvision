## Documented intent (from 6.1)

All three ports' docs collectively promise:

> `getTimestampSeconds()` returns `T_capture`, the moment the camera shutter opened, in
> the robot's FPGA-time seconds, suitable to pass directly to
> `SwerveDrivePoseEstimator.addVisionMeasurement`.

## Implementation reality (synthesis of 4.2 + 5.2)

| Port   | Formula                                | Numerical result            | Verdict          |
| ------ | -------------------------------------- | --------------------------- | ---------------- |
| Java   | `captureTimestampMicros / 1e6`         | `T_capture` (within µs)     | matches intent ✓ |
| Python | `(ntReceiveTimestamp − latency) / 1e6` | `T_capture + network_delay` | matches intent within network jitter (ms) ✓ |
| C++ `GetAllUnreadResults` | `(value.time − latency) − latency` | `T_capture − latency`       | **off by ~one full latency (~16 ms @ 60 fps)** — BLOCKER (5.2) |
| C++ `GetLatestResult` (deprecated) | `now − latency`         | `T_capture − network_delay` | matches intent within network jitter; correct ✓ |

## Cross-port comparison test

A robot subscribes to a single PhotonVision result on NT. For the same wire bytes:

- A Java consumer sees `getTimestampSeconds()` ≈ T_capture.
- A Python consumer sees `getTimestampSeconds()` ≈ T_capture + 2 ms (network delay).
- A C++ consumer using the **modern** API sees `GetTimestamp()` ≈ T_capture − 16 ms.

That 16 ms gap is well within wpilib's vision-history buffer (~1.5 s), so the C++ value
is **accepted by the pose estimator** but applied against odometry **one frame in the
past**. The C++ robot's vision-fused pose systematically lags.

## Already-filed findings covering this stage

1. **[BLOCKER] 5.2** — C++ `GetAllUnreadResults` double-subtracts latency.
2. **[DIVERGENCE] 4.2** — Java vs C++/Python use different formulas (now superseded by
   the BLOCKER framing, per the correction filed at 5.2).
3. **[SMELL] 5.1** — Transient `getOffset() == 0` during startup / NT reconnect
   contaminates captureTimestampMicros.
4. **[DIVERGENCE] 1.1** + correction at 4.2 — Java/Python doc comments still claim
   "coprocessor's time base" when the wire value is server-timebase µs.

## Adjacent gaps surfaced during this verification (not new findings)

- **Java's `getLatencyMillis()` is deprecated** in PhotonPipelineResult.java:111-115 but
  still called from PhotonPipelineMetadata.java:56-58. Even though the field is also
  `@Deprecated`, NTDataPublisher publishes the result of `getLatencyMillis()` to the
  `latencyMillisEntry` NT topic on every frame. Backward-compat surface for legacy
  clients; harmless until removed.
- **Python's `getTimestampSeconds` references `self.ntReceiveTimestampMicros` which has
  no `metadata.` prefix** — i.e., it's a sibling field of `metadata`, not a metadata
  field. The contract is that the receiver populates this locally per packet. Clear in
  the dataclass definition, but a reader following the Java path naively might miss
  this. Doc clarity, not a bug.
- **`getLatestResult` is deprecated in all three ports** but still callable. C++'s
  deprecated path happens to be the only correct one due to the 5.2 BLOCKER. A user who
  worked around the bug by using the deprecated API would have correct behavior — until
  the deprecated API is removed.

## Outcome

**No new finding for 6.2.** This check is the verification step against the documented
intent established in 6.1. The verification result is: **C++'s modern API does NOT
match the documented intent** (off by ~one latency, the 5.2 BLOCKER); Java and Python
match within their respective error bounds.

The 4.2 DIVERGENCE + 5.2 BLOCKER + 5.1 SMELL + 1.1 DIVERGENCE collectively encode the
implementation-vs-intent mismatch for `getTimestampSeconds`. No additional finding is
needed here.

Stage 6 complete.

## For Stage 7+

The pose-strategy math (Stage 7) interacts with `getTimestampSeconds` because every
`EstimatedRobotPose` carries a timestamp from this method. A C++ robot using ANY
PoseStrategy gets timestamps that are one frame in the past. The 5.2 BLOCKER thus
contaminates the entire C++ pose-estimation pipeline, not just one particular strategy.
This should be remembered when filing findings for Stage 7 strategies.
