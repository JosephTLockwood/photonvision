## The transient-zero issue

The same return value (`0`) means three different things depending on context:

| State                                  | Meaning                          | `getOffset()` returns |
| -------------------------------------- | -------------------------------- | --------------------- |
| Server mode (robot)                    | Server timebase == local         | `0` (correct)         |
| Client mode, fresh JNI client          | No pongs received yet            | likely `0` (unknown)  |
| Client mode, after `reportNtConnected` | New JNI client; no pongs yet     | likely `0` (unknown)  |
| Client mode, after first round-trip    | Real offset (signed µs)          | non-zero (correct)    |
| Client + server both null              | (unreachable)                    | `0` (error)           |

`NTDataPublisher.java:186` unconditionally adds `offset` to local capture time. During the
transient "no-pong-yet" windows, the published `captureTimestampMicros = capture_local +
0 = capture_local` — which is in the **coprocessor's timebase**, not the server's. A
robot consumer reading the value expecting server-timebase µs gets a value that's off by
"server boot time relative to coproc boot time" — typically tens of seconds.

### How wide is this window?

- **At startup**: the constructor at line 58 creates the client pointing at `127.0.0.1`.
  Local-loopback pings/pongs would converge near-instantly, but offset to localhost is
  meaningless. The first `tick()` (up to 1000 ms later) sees `ntInstance.getConnections()`
  and re-targets to the real server. After `setServer()`, the JNI client must do at least
  one full ping-pong roundtrip to update its cached offset. Total window: ~1–2 s after
  PhotonVision startup.
- **After NT reconnect**: `reportNtConnected()` stops + recreates the client. The new
  client starts with no pongs → offset unknown. Until the first roundtrip after the next
  `tick()` re-targets it, offset is uncertain. Window: ~1–2 s after every NT reconnect.

### Downstream impact

WPILib's `SwerveDrivePoseEstimator.addVisionMeasurement` (and similar) typically discard
vision measurements older than `~1.5 s` (the default history buffer size). If the
transient-zero offset translates a coproc-time measurement into the past beyond that
window, the measurement is silently dropped — robot doesn't observe the bug.

But if the coproc-vs-server offset is **less than 1.5 s** in absolute value (e.g., the
coproc and server both booted close together), the bad-offset measurement passes the
staleness filter and gets blended into the pose estimate with a wrong timestamp. Pose
estimate jumps briefly after every NT reconnect, then settles. Could look like
intermittent vision-pose hiccups.

## Why I cannot fully verify

The JNI's exact "no pongs yet" return value is implementation-defined. If
`TimeSyncClient.getOffset()` returns `Long.MIN_VALUE` (or any sentinel) before convergence,
the issue is contained at the JNI boundary. The Java code in PhotonVision doesn't check
for any sentinel — it trusts whatever the JNI returns and adds it directly to capture
timestamps. So even if the JNI signaled "not converged", PhotonVision wouldn't notice.

Filing as **SMELL** with **low-medium confidence**: the gap exists in code clearly; the
real-world impact depends on JNI behavior and field timing (boot offsets, network
delay, NT reconnect frequency).
