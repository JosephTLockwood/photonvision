## The canonical pattern (Java + C++)

Both Java and C++ poseest examples follow the same structure:

    for (var result : camera.getAllUnreadResults()) {
        visionEst = photonEstimator.estimateCoprocMultiTagPose(result);
        if (visionEst.isEmpty()) {
            visionEst = photonEstimator.estimateLowestAmbiguityPose(result);
        }
        var estStdDevs = getEstimationStdDevs(...);
        estConsumer.accept(est.estimatedPose.toPose2d(),
                          est.timestampSeconds,           // ← passed directly
                          estStdDevs);
    }

`estConsumer` ultimately calls `SwerveDrivePoseEstimator.addVisionMeasurement(pose,
timestampSeconds, stddev)`. WPILib's contract for `timestampSeconds` is "FPGA timestamp
in seconds, matching `Timer.getFPGATimestamp()`".

## Stddev heuristic (Java + C++, identical structure)

Both ports compute stddev via the same heuristic:
- Default: `kSingleTagStdDevs` (a 3-vector for x/y/heading).
- If multi-tag: use `kMultiTagStdDevs` (smaller, trusted more).
- If single-tag at >4m: set to `Double.MAX_VALUE` (effectively reject the measurement).
- Otherwise scale: `estStdDevs * (1 + avgDist^2 / 30)`.

This matches the recipe inventoried in check 8.2. Robot authors copy this pattern
verbatim into their own code.

## Python example

[drivetrain.py:139-143](photonlib-python-examples/poseest/drivetrain.py#L139-L143):

    def addVisionPoseEstimate(self, pose: Pose3d, timestamp: float) -> None:
        self.poseEst.addVisionMeasurement(pose, timestamp)

Python passes only `(pose, timestamp)` — no stddev. WPILib's Python pose estimator
applies default stddev internally. This is a simplification, not a bug; it just means
the Python example trusts vision more (or less, depending on the default) than the
Java/C++ examples.

## Implication: 5.2 BLOCKER lands on the C++ example users

The Java and C++ examples both assume `est.timestampSeconds` is FPGA-relative. Per the
verification at check 6.2:

- **Java**'s `est.timestampSeconds` = `captureTimestampMicros / 1e6` = correct FPGA time
  on the robot (server-timebase ≡ FPGA-time when robot runs the timesync server).
- **C++**'s `est.timestamp` = `value.time - 2*latency` per the 5.2 BLOCKER = `T_capture
  - latency` seconds. **Off by one latency** (~16 ms at 60 fps).

So a robot team using the C++ poseest example **as-is** gets vision measurements
timestamped ~16 ms earlier than the actual capture moment. The
SwerveDrivePoseEstimator then interpolates against the odometry sample one frame in the
past, leading to a systematic ~one-frame lag in the vision-fused pose.

This **does not require any user mistake** — it's a direct consequence of the 5.2
BLOCKER affecting the documented example pattern. C++ users following the example
exactly inherit the bug.

## Adjacent observations (not new findings)

- The example's `1 + avgDist^2 / 30` formula treats distance as the dominant
  uncertainty driver. This is reasonable PnP physics (uncertainty scales with
  distance squared) but the constant 30 is empirical and may need tuning per-camera.
- The example's `Double.MAX_VALUE` rejection for single-tag-at-4m is a "soft reject"
  that doesn't fully zero out the contribution — it just makes the Kalman gain
  vanishingly small. Mathematically equivalent to rejection in most filters but
  could cause numerical issues if the underlying matrix-math implementation has
  divide-by-near-zero behavior. NIT.
- The example doesn't show users how to handle the `est.strategy` field
  (it's not consulted). Combined with the 7.1 DIVERGENCE (Python lacks the strategy
  field), the example pattern works the same on all ports — strategy is invisible.
  Documentation gap: a stddev heuristic that branches on strategy (e.g., trust
  CONSTRAINED_SOLVEPNP less than MULTI_TAG_PNP_ON_COPROCESSOR) would surface the field's
  usefulness.

## Outcome

No new finding for 9.1. The check confirms:
1. Example projects exist for Java, C++, and Python.
2. They all pass `est.timestampSeconds` (or equivalent) **directly** to
   `addVisionMeasurement`, assuming it's FPGA-relative.
3. Java + Python examples are correct because their timestamps are FPGA-relative.
4. **C++ example silently inherits the 5.2 BLOCKER** — every example user gets
   ~16-ms-too-early vision measurements.

This dramatically increases the user-visible blast radius of the 5.2 BLOCKER:
the bug isn't just lurking in the library, it's actively being demonstrated as
the canonical pattern in the official C++ example.
