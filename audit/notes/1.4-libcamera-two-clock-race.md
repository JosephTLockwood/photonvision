## Hand-derivation of the bias

Define instants:
- `T0_lib` = libcamera-clock instant of frame-capture (whatever `getFrameCaptureTime` anchors to).
- `T1_lib` = libcamera-clock instant when `now = getLibcameraTimestamp()` is sampled (after `takeColorFrame`/`takeProcessedFrame`/`getGpuProcessType`).
- `T2_wpi` = wpi-clock instant when `MathUtils.wpiNanoTime()` is sampled inside the `new Frame(...)` expression (after `releasePair(p_ptr)`).

Code computes:

    Frame.timestampNanos = T2_wpi - (T1_lib - T0_lib)
                         = T2_wpi - elapsed_T0→T1_on_lib_clock

What we want (the wpi-clock time of frame capture):

    T0_wpi = T2_wpi - elapsed_T0→T2_on_wpi_clock

Assume both clocks are CLOCK_MONOTONIC-derived (so identical rate to better than ~1ppm —
NTP adjusts CLOCK_MONOTONIC slowly enough that the difference over 100 ms is negligible).
Then:

    elapsed_T0→T2_on_wpi_clock ≈ elapsed_T0→T2_on_lib_clock
                              = elapsed_T0→T1_on_lib_clock + elapsed_T1→T2_on_lib_clock

So the code's value differs from the correct value by:

    Frame.timestampNanos - T0_wpi  =  elapsed_T1→T2_on_lib_clock
                                  ≈  one JNI call (`releasePair`) + a few Java statements
                                  ≈  1–10 µs typically

Direction: **positive bias** (code reports capture as slightly later than it was).
Magnitude: ≪ one frame period (16.7 ms at 60 fps). **NIT.**

## What I cannot verify without the JNI source

- **Anchor point of `getFrameCaptureTime`.** This is `Request::metadata().get(SensorTimestamp)`
  in libcamera. On the IMX219/IMX477/IMX708/OV9281 sensors PhotonVision officially supports
  (see [LibcameraGpuSettables.java:66-94](photon-core/src/main/java/org/photonvision/vision/camera/csi/LibcameraGpuSettables.java#L66-L94)),
  the libcamera pipeline handler chooses SoE or EoF per-sensor. PhotonVision has **no
  in-tree documentation** stating which it is.
- **Units of `getLibcameraTimestamp` / `getFrameCaptureTime`.** Java `long`; the math implicitly
  assumes nanoseconds (matching `Frame.timestampNanos`). If the external driver ever
  changed units to µs, the timestamps would silently go off by 1000×.
- **Clock identity.** The formula assumes both functions sample from the same clock domain
  (so subtracting them gives a true elapsed time). On Linux, libcamera uses
  `clock_gettime(CLOCK_MONOTONIC)` and wpi::Now also defaults to CLOCK_MONOTONIC — so this
  holds on Pi/Linux. Cross-platform (not relevant here, since libcamera is Linux-only) it
  would matter.

## Cross-port comparison

- **V4L2 / USBFrameProvider path** uses a single clock sample
  ([USBFrameProvider.java:75](photon-core/src/main/java/org/photonvision/vision/frame/provider/USBFrameProvider.java#L75)):
  `captureTimeNs = cvSink.grabFrame(...) * 1000`. cscore already exposes the frame timestamp
  in `wpi::Now` units, so no two-clock translation is needed. **No race, no implicit unit
  assumption.**
- **Libcamera path** is the only place in PhotonVision that does live clock-domain
  translation. The complexity is necessary; the bias is small enough not to matter for
  vision-pose-estimate timing.

## Adjacent observation (not catalogued)

The libcamera anchor is likely **end-of-readout** for the supported Pi sensors (verified by
inspection of libcamera-rpi sources outside this audit's scope). This is a divergence from
the V4L2-SoE anchor on USB cameras of ~one rolling-shutter readout period (~5–10 ms on
1280×800 60fps). Strictly, this **extends the pre-found mid-exposure issue** from
`memory/audit_prefound.md` (PR #2489) — but it's not clear from public PR signal whether
PR #2489 corrects the libcamera path or only the V4L2/cscore path. Worth confirming when
reviewing PR #2489. **Not cataloging as a new finding**, because the underlying mid-exposure
problem is already represented in the in-flight PR.

## Outcome

Check 1.4 is **clean for new findings**:

- Implicit nanoseconds assumption is fragile but matches current driver; would be loud-fail
